# Reliability

이 문서는 `AI-Product-Factory` 저장소의 신뢰성 기준을 정의한다.
에이전트가 코드를 변경할 때 서비스의 안정성을 유지하기 위한 규칙이다.

---

## 핵심 원칙

- **검증 없이 배포하지 않는다.** 모든 코드 변경은 `npm run verify` 통과 후에만 커밋한다.
- **깨진 것을 더 깨뜨리지 않는다.** 기존 검증이 실패 중이면, 먼저 실패를 해결하거나 인간에게 보고한다.
- **작게 바꾸고, 자주 확인한다.** 큰 변경보다 작은 PR을 여러 개 만든다.
- **롤백 가능성을 항상 유지한다.** 되돌릴 수 없는 변경은 인간 승인 후에만 진행한다.

---

## 검증 파이프라인

### 필수 검증 흐름

모든 active 서비스는 동일한 3단계 파이프라인을 따른다:

```
lint (check-syntax.mjs) → test (node --test) → build
```

| 서비스 | 명령 | 상태 |
|--------|------|------|
| IBAD/app | `cd IBAD/app && npm run verify` | active |
| Translate-Developer | `cd Translate-Developer && npm run verify` | active |
| Spending-Personality | `cd Spending-Personality && npm run verify` | active |
| UGGK | 없음 | spec-first (디렉토리 미생성) |

### 검증 규칙

| 규칙 | 설명 |
|------|------|
| 변경 전 검증 | 코드 수정 전 현재 상태의 검증을 먼저 실행하여 기존 실패 여부를 확인한다 |
| 변경 후 검증 | 모든 수정이 끝난 후 반드시 검증을 실행한다 |
| 커밋 전 통과 필수 | 검증 실패 상태에서 커밋하지 않는다 |
| 실패 시 중단 | 검증이 실패하면 수정하거나, 수정 불가 시 인간에게 보고한다 |
| 범위 한정 검증 | 변경한 서비스의 검증만 실행한다. 관련 없는 서비스는 실행하지 않는다 |

### 문서 전용 변경

코드가 아닌 문서만 변경한 경우:
- 자동화된 검증이 없으면 그 사실을 명시한다.
- 링크 유효성, 문서 간 정합성을 수동으로 확인한다.
- 향후 PR 0-15(doc-gardening/lint)에서 자동 검증을 추가한다.

## 배포 안전성

### Vercel 배포 모델

모든 서비스는 Vercel에 배포된다. PR 병합 시 자동 배포가 트리거된다.

| 규칙 | 설명 |
|------|------|
| 검증 통과 필수 | 검증 실패한 PR은 병합하지 않는다 |
| 단일 서비스 변경 | PR당 하나의 서비스만 변경하여 배포 영향 범위를 제한한다 |
| 환경 변수 금지 | 비밀, 토큰, API 키를 리포에 저장하지 않는다 |

### 배포 전 체크리스트

에이전트가 PR을 생성하기 전 확인할 항목:

- [ ] `npm run verify` 통과
- [ ] 변경한 서비스 외 다른 서비스에 영향 없음 확인
- [ ] 환경 변수나 비밀이 코드에 하드코딩되지 않음
- [ ] 새 의존성 추가 시 `package.json`에만 선언 (`package-lock.json` 동기화)

## 오류 처리 원칙

### 사용자 대면 오류

| 원칙 | 설명 |
|------|------|
| 친절한 메시지 | 기술 용어 대신 사용자가 이해할 수 있는 한국어로 표시한다 |
| 행동 안내 | "오류가 발생했습니다"보다 "다시 시도해 주세요"처럼 다음 행동을 안내한다 |
| 정보 노출 금지 | 스택 트레이스, 내부 경로, API 키를 사용자에게 노출하지 않는다 |

단, 오류 메시지 문구 자체는 제품 결정이다. PRD에 명시된 것을 쓰고, 없으면 [`PRODUCT_SENSE.md`](PRODUCT_SENSE.md)의 규칙에 따라 인간에게 질문한다.

### 외부 API 오류

| 상황 | 행동 |
|------|------|
| 외부 API 일시 장애 | 재시도 로직이 있으면 사용, 없으면 사용자에게 안내 |
| 외부 API 응답 변경 | 기존 계약과 다른 응답이면 중단하고 인간에게 보고 |
| API 키 만료/무효 | 코드로 해결 불가. 인간에게 보고 |

## 롤백 기준

### 롤백이 필요한 상황

| 상황 | 행동 |
|------|------|
| 배포 후 서비스 불능 | 이전 커밋으로 revert PR 생성 |
| 배포 후 핵심 기능 장애 | 인간에게 보고 + revert PR 준비 |
| 배포 후 경미한 문제 | 수정 PR로 대응 (롤백 불필요) |

### 롤백 규칙

- revert PR도 검증 파이프라인을 통과해야 한다.
- 롤백 판단은 에이전트가 할 수 있으나, 실행 전 인간에게 보고한다.
- 긴급 상황(서비스 불능)에서는 보고와 동시에 revert PR을 생성한다.

## 되돌릴 수 없는 변경

다음은 인간 승인 없이 실행하지 않는다:

| 변경 유형 | 위험 |
|-----------|------|
| 데이터 스키마 변경 | 기존 데이터 호환성 깨짐 가능 |
| 외부 API 계약 변경 | 클라이언트 호환성 깨짐 가능 |
| 주요 의존성 메이저 버전 업그레이드 | 하위 호환성 깨짐 가능 |
| 서비스 디렉토리 구조 변경 | 배포 설정, 라우팅 깨짐 가능 |
| `git push --force`, `git reset --hard` | 히스토리 손실 |

## 서비스별 신뢰성 요구사항

현재는 모든 서비스에 동일한 기준을 적용한다. 서비스별 예외가 필요하면 이 섹션에 추가한다.

| 서비스 | 추가 요구사항 |
|--------|-------------|
| IBAD/app | 없음 (공통 기준 적용) |
| Translate-Developer | 없음 (공통 기준 적용) |
| Spending-Personality | 없음 (공통 기준 적용) |
| UGGK | spec-first 상태이므로 코드 신뢰성 기준 미적용 |

## Fail-Closed 기본값

- 검증 실행 불가(환경 문제 등)이면 커밋하지 않는다.
- 검증 결과가 모호하면 실패로 취급한다.
- 롤백 필요성이 모호하면 인간에게 보고하고 대기한다.
- 배포 영향 범위가 불확실하면 단일 서비스 범위로 제한한다.
