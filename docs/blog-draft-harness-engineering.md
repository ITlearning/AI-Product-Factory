# 사고 흐름 노트 (글 쓸 때 참고용)

> 이 섹션은 블로그 초안이 아닙니다. 그때 내 생각이 어떻게 흘러갔는지를 기록한 메모입니다.
> 초안은 아래 구분선 이후에 있습니다.

---

## 1. 출발점: 꿈이 있었다

처음 `AI-Product-Factory`를 시작할 때 머릿속에 있던 문장:

> *"기획서만 던져주면 하네스 & 멀티에이전트로 프로젝트를 만든다."*

이게 이 저장소의 원래 취지였다. 틀린 말이라고 생각하지 않았다. 근데 막상 설계를 시작하니까 이 문장을 **운영 계약으로 쓰기엔 너무 강하다**는 느낌이 왔다. "기획서만 던지면"에서 "기획서"가 정확히 뭔지, 누가 검증하는지, 에이전트가 잘못 이해하면 어떻게 멈추는지가 하나도 없었다.

---

## 2. 첫 번째 전환: PRD는 인간이 소유해야 한다

AI가 제품 결정을 대신하면 안 된다는 게 점점 명확해졌다.

> *"기획적인 요소는 인간이 많이 개입해야 할 것 같아. 어디까지나 AI는 내가 원하는 프로젝트를 만들어주는 역할을 해야 할 것 같고."*

이 말을 하면서 PRD 소유권이 확실히 정리됐다. 목표, 비목표, 성공 기준, 우선순위, 범위 — 이건 에이전트가 채워주는 게 아니라 내가 먼저 결정하고 잠가야 할 것들이었다. 에이전트는 그 이후를 맡는다.

---

## 3. 멀티 에이전트로 설계를 비평받았다

설계를 혼자 밀지 않고 여러 역할(critic, architect, analyst, writer)로 반박을 받아봤다.

그 중 가장 인상 깊었던 결론:

> *"review gate와 evidence trail이 없으면 멀티 에이전트는 빨라질 뿐, 일관되게 틀릴 수 있다."*

이게 Sprint 0의 목표를 바꿨다. 처음엔 "뭔가 만들어지는 것"을 목표로 했는데, **잘 멈추는 것을 먼저 설계**하는 방향으로 전환했다.

---

## 4. Ralph 루프는 포함되는가?

중간에 확인하고 싶었던 것:

> *"랄프 루프 방식도 포함되어있는지 확인해줘."*

결론: 포함된다. 근데 위치가 중요했다. PRD 앞에 붙이는 게 아니라, **spec-lock 이후의 실행 루프**에 붙는다. 순서가 있었다.

---

## 5. 완벽하게 시작하지 않기로 했다

설계가 너무 커질 것 같다는 느낌이 왔을 때 내가 한 말:

> *"초반부터 완벽하게 만들 수는 없으니까 만들어가면서 고도화시키자."*

이게 Sprint 0을 "기능 개발 없이 문서만 설치"로 확정한 계기였다. 잘 만드는 것보다 잘 멈추는 것. 실패 지점이 드러나면 그게 다음 개선의 입력이 된다.

---

## 6. Sprint 0이 끝난 지금

15개의 PR. 코드는 한 줄도 없다. 전부 규칙, 계약, 게이트, 포맷이다.

이걸 "아무것도 안 한 것"처럼 느낄 수도 있는데, 나는 이게 **가장 중요한 스프린트**였다고 생각한다. 안전망 없이 에이전트를 풀어놓으면 빠르게 틀린 방향으로 달려갈 뿐이니까.

---

## 7. 딸-깍은 어디서부터 가능한가

설계를 마치면서 깨달은 것:

> *"딸-깍이 가능한 구간과 불가능한 구간이 다르다."*

- **딸-깍 불가**: 아이디어 → 기획 → PRD → spec-lock. 여기는 인간의 사고와 판단이 필요하다. AI가 기획 초안을 잡아주면 편해 보이지만, 그 순간 방향을 AI가 정한 것이 된다.
- **딸-깍 가능**: spec-lock 이후. 잠긴 계약 위에서 ralph가 돌아가는 구간은 실질적으로 딸-깍이다.

결국 이 저장소의 진짜 슬로건은:

> **인간이 PRD를 잠그면, 나머지는 딸-깍이다.**

# 하네스 엔지니어링 적용기: AI 프로덕트 팩토리에 안전망을 깔다

> "기획서만 던져주면 AI가 다 만든다" — 이 꿈은 틀리지 않았지만, 그 전에 먼저 해야 할 일이 있었다.

---

## 시작은 하나의 질문이었다

`AI-Product-Factory`라는 저장소를 운영하고 있다. 처음 목표는 단순했다. 인간이 기획만 하면 하네스와 멀티 에이전트가 알아서 분해하고 구현해 PR까지 올려주는 것.

막상 시작하니 문제가 보였다. **에이전트에게 "만들어" 하고 던지면 뭔가는 만들어진다. 근데 내가 원하는 게 맞냐고.** 잘못 만들어졌을 때 어떻게 세우냐고.

그게 출발점이었다.

---

## 하네스 엔지니어링이란

[OpenAI의 하네스 엔지니어링 글](https://openai.com/ko-KR/index/harness-engineering/)을 읽고 나서 핵심을 이렇게 잡았다.

하네스 엔지니어링은 **"더 많이 생성하기"가 아니라 "잘못된 생성을 빨리 막기"를 먼저 설계하는 것**이다.

에이전트가 코드를 빠르게 쓰는 건 이미 된다. 문제는 그 코드가 원하는 방향인지, 기획을 벗어나진 않는지 보장할 구조가 없다는 거다. 하네스는 그 구조다.

---

## 잠근 것과 열어둔 것

설계를 시작할 때 가장 먼저 한 게 경계 긋기였다. 인간이 소유할 것과 에이전트가 소유할 것.

**인간 소유**

- PRD 작성
- 목표 / 비목표 결정
- 성공 기준 확정
- 우선순위와 범위
- spec-lock 승인
- stop 조건 지정

**에이전트 소유**

- PRD를 실행 계약으로 정리
- 스프린트/PR 단위 분해
- 코드·테스트·문서 작성
- 검증과 증거 수집
- 리뷰 대응
- PR handoff

"기획서만 던지면 된다"는 말이 너무 강하다는 걸 논의하면서 깨달았다. PRD는 인간이 직접 잠가야 한다. 에이전트는 그 이후 실행 루프를 맡는다.

---

## 공식 실행 경로

가장 공들인 부분이 **경로**다.

```
PRD → admission → spec-lock → ralplan → sprint/PR 분해 → ralph 실행 → 리뷰 점수 게이트
```

| 단계 | 역할 |
|------|------|
| **PRD** | 인간이 작성하는 제품 요구사항 |
| **admission** | PRD가 실행 가능한 계약인지 검증. 목표·비목표·성공 기준 없으면 자동 반려 |
| **spec-lock** | 인간이 잠금 승인. 이후엔 무단 수정 불가 |
| **ralplan** | 모호한 실행 요청을 계획 단계로 우회. 스프린트/PR 단위로 분해 |
| **ralph 실행** | 구현 → 검증 → 리뷰 → 수정 → 재검증 반복 |
| **리뷰 점수 게이트** | critic·architect·analyst 리뷰 통과 후 인간 최종 승인 |

이 경로는 건너뛰거나 순서를 바꿀 수 없다. **spec-lock 전 실행 금지**가 첫 번째 금지 조항이다.

---

## 멀티 에이전트로 설계를 검증했다

혼자 밀지 않았다. critic·architect·analyst·writer 역할로 반박을 받으면서 다듬었다.

가장 날카로웠던 결론:

> *"review gate와 evidence trail이 없으면 멀티 에이전트는 빨라질 뿐, 일관되게 틀릴 수 있다."*

이 한 문장이 Sprint 0의 방향을 바꿨다. **잘 생성하기보다 잘 멈추기를 먼저 설계한다.**

---

## Sprint 0: 하네스 문서 기반 설치

Sprint 0은 기능 개발이 아니었다. **하네스를 까는 스프린트**였다.

15개 PR. 코드 한 줄 없이 문서만 만들었다.

| 문서 | 역할 |
|------|------|
| `CHARTER.md` | 책임 경계·실행 경로·중단 규칙·리뷰 게이트 헌장 |
| `AGENTS.md` | 저장소 진입점 맵 (백과사전이 아니라 맵) |
| `ARCHITECTURE.md` | 기술 아키텍처 |
| `PLANS.md` | 계획 관리 규칙 |
| `QUALITY_SCORE.md` | 리뷰 점수 계산 구조 (역할별 최소 7/10, 평균 8.0 이상) |
| `PRODUCT_SENSE.md` | 제품 감각 가이드 |
| `RELIABILITY.md` | 신뢰성 기준 |
| `SECURITY.md` | 보안 기준 |
| `admission.md` | PRD 실행 허가 계약 템플릿 |
| `spec-lock.md` | 잠금 규칙 |
| `decomposition.md` | Sprint/PR 분해 수치 기준 |
| `ralplan-ralph.md` | ralplan → ralph 공식 경로 |
| `evidence-trail.md` | 검증 증거 포맷 |
| `doc-lint.md` | 문서 전용 변경 검증 규칙 |

규칙과 계약과 게이트만 먼저 설치했다.

## AGENTS.md에 대한 생각

처음엔 `AGENTS.md`를 크고 상세하게 만들려 했다. 근데 그러면 안 된다는 결론이 났다.

> **AGENTS.md는 백과사전이 아니라 맵이다.**

크게 만들면 컨텍스트 낭비, 오래된 규칙 누적, 뭐가 중요한지 흐려짐, 기계적 검증 어려움. 사단이 난다.

`AGENTS.md`는 어디를 봐야 하는지만 알려주고, 실제 규칙은 `docs/` 아래에 있다.

## 설계 철학: 잘 멈추는 것을 먼저

이 하네스의 1차 목표는 두 가지다.

- **Fail-Closed**: 판단 안 되면 허용 말고 차단
- **잘 멈추기**: 애매하면 중단. 검증·리뷰 기록이 있어야만 진행

에이전트가 즉시 멈춰야 하는 조건도 6개 박아뒀다. 기획 이탈, 모호성 해소 불가, 범위 확대 유혹, 외부 의존성 장벽, 리뷰 점수 미달, blocker 발견.

---

## 아이디어·기획·PRD는 인간의 영역이다

설계를 마무리할 때 한 가지 질문이 걸렸다. *"AI랑 같이 기획을 다듬으면 안 되나?"*

같이 다듬는 건 가능하다. 근데 경계를 잘못 그으면 하네스가 느슨해진다.

토스 테크 블로그에 이런 말이 있다.

> *"차이는 단 하나, 작업 전에 컨텍스트를 설계했느냐의 여부"*

컨텍스트를 설계하는 건 인간이다. AI가 기획 초안을 잡아주면 인간은 "검토"하는 역할로 밀려난다. 형식적으로 확정은 인간이 했어도 실질적으로 방향은 AI가 정한 거다. 그 순간 PRD의 소유권이 흐려진다.

단계별로 정리하면 이렇다.

| 단계 | 인간 | AI |
|------|------|-----|
| 아이디어 | 주도 | 없음 |
| 기획 | 주도 | 없음 |
| PRD 작성 | 주도 | 없음 |
| PRD 검증 (admission) | 승인 | 빠진 필드 지적·구조화 보조 |
| spec-lock | 확정·잠금 | 없음 |
| 실행 (ralplan → ralph) | 승인·중단 | 주도 |

AI가 PRD에서 할 수 있는 건 내용을 채우는 게 아니다. 빠진 걸 지적하는 거다. 목표가 없으면 목표를 써주는 게 아니라 "목표가 없습니다"라고 말하는 것.

토스 글의 **Executable SSOT** 개념도 같은 맥락이다. 문서가 사람에게는 가이드라인이고 AI에게는 시스템 프롬프트가 되려면 그 문서를 사람이 설계해야 한다. AI가 설계한 컨텍스트 위에서 AI를 돌리면 순환이다.

---

## 딸-깍은 어디서부터 가능한가

AI 시대에 자주 나오는 말. **딸-깍**. 클릭 한 번으로 모든 게 돌아간다는.

근데 딸-깍이 가능한 구간과 불가능한 구간이 다르다.

```
아이디어 → 기획 → PRD → spec-lock        ← 딸-깍 불가. 인간의 판단 구간.
spec-lock → ralplan → ralph → 리뷰 게이트  ← 딸-깍 가능. 잠긴 계약 위에서 돌아간다.
```

기획 단계에서 딸-깍을 시도하면 AI가 방향을 정하게 된다. 인간이 승인하는 형식이어도 PRD 안에 AI 판단이 들어간 거다. 그걸 잠그고 실행하면 내가 원하는 게 아니라 AI가 생각한 게 나온다.

그래서 슬로건이 이렇게 됐다.

> **인간이 PRD를 잠그면, 나머지는 딸-깍이다.**

"기획서만 던지면 된다"가 너무 강했던 이유다. 던지는 행위 자체는 인간의 일이다. 딸-깍은 던진 이후부터.

---

## 다음 단계

Sprint 0이 끝났다. 하네스 문서는 갖춰졌고, 이제 PRD를 작성해서 이 경로를 실전에 돌릴 차례다.

"기획서만 던져주면 된다"는 꿈은 여전히 목표다. 다만 그 전에 **던질 기획서가 올바른 계약인지 검증하는 게이트**를 먼저 만들었다. 그게 Sprint 0의 전부였다.
