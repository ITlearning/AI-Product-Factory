# Product Sense

이 문서는 `AI-Product-Factory` 저장소에서 에이전트가 제품 관련 판단을 만날 때의 행동 규칙을 정의한다.
에이전트에게 제품 감각을 부여하는 것이 아니라, **제품 판단이 필요한 순간을 인식하고 올바르게 위임하는 규칙**을 정의한다.

---

## 핵심 원칙

- 에이전트는 **제품 결정을 대신하지 않는다**.
- 에이전트는 제품 결정이 필요한 순간을 **인식**하고 **인간에게 위임**한다.
- 제품 판단과 기술 판단의 경계가 모호하면 **제품 판단으로 취급**한다 (fail-closed).
- PRD에 명시된 것만 구현한다. PRD에 없는 제품 개선을 자의적으로 추가하지 않는다.

---

## 제품 결정 vs 기술 결정

| 구분 | 제품 결정 (인간 소유) | 기술 결정 (에이전트 자율) |
|------|----------------------|------------------------|
| 정의 | 사용자에게 보이는 것, 느껴지는 것, 제공되는 가치에 영향을 주는 결정 | 사용자에게 보이지 않는 구현 방식, 구조, 성능에 관한 결정 |
| 예시 | 기능 추가/제거, UI 텍스트 변경, 사용자 흐름 변경, 오류 메시지 문구, 기본값 설정 | 함수 분리, 파일 구조, 변수명, 빌드 최적화, 리팩토링 |
| 경계 사례 | 성능 저하가 사용자 체감에 영향 → 제품 결정 | 성능 개선이 사용자 체감에 무관 → 기술 결정 |
| 모호할 때 | **인간에게 질문** | — |

### 판정 체크리스트

다음 중 하나라도 "예"이면 제품 결정이다. 에이전트는 자율적으로 결정하지 않고 인간에게 위임한다.

- [ ] 사용자가 직접 보거나 상호작용하는 것이 바뀌는가?
- [ ] 사용자에게 제공되는 가치가 바뀌는가?
- [ ] PRD에 명시되지 않은 새로운 행동이 추가되는가?
- [ ] 기존 사용자 기대를 깨뜨릴 수 있는가?
- [ ] 다른 서비스나 기능에 연쇄 영향을 줄 수 있는가?

## 에이전트의 제품 관련 행동 규칙

### 해야 하는 것

| 상황 | 행동 |
|------|------|
| PRD에 명시된 기능 구현 | 그대로 구현한다 |
| 구현 중 제품 결정이 필요한 분기 발견 | 중단하고 인간에게 선택지를 제시한다 |
| PRD의 요구사항이 모호함 | 해석하지 않고 인간에게 질문한다 |
| 사용자 대면 텍스트(UI, 오류 메시지) 작성 필요 | PRD에 명시된 것만 쓰고, 없으면 인간에게 질문한다 |
| 기존 기능에 영향을 줄 수 있는 변경 | 영향 범위를 먼저 보고한다 |

### 하지 말아야 하는 것

| 금지 행동 | 이유 |
|-----------|------|
| "이게 더 나을 것 같아서" 기능 추가 | 제품 판단은 인간 소유 |
| PRD에 없는 UX 개선 자의적 적용 | 범위 확대, spec-lock 위반 |
| 사용자 흐름 임의 변경 | 제품 결정 |
| 오류 메시지/안내 문구 임의 작성 | 사용자 대면 텍스트는 제품 영역 |
| "사용자가 원할 것 같다"는 추측 기반 진행 | source-of-truth 밖 추측 진행 금지 |

## 제품 품질 지표

에이전트가 제품 결정을 하지는 않지만, 다음 지표를 인식하고 위반을 보고해야 한다.

### 기본 품질 체크

| 지표 | 에이전트 행동 |
|------|-------------|
| 기능 완전성 | PRD의 성공 기준이 모두 충족되는지 확인 |
| 일관성 | 같은 서비스 내에서 UI 패턴, 용어, 동작이 일관된지 확인 |
| 오류 처리 | 사용자에게 노출되는 오류가 친절하고 명확한지 확인 |
| 접근성 | 명백한 접근성 문제(빈 alt 태그, 키보드 네비게이션 불가 등)를 보고 |

### 보고 의무

에이전트는 다음을 발견하면 수정하지 않고 인간에게 보고한다:

- PRD와 현재 구현 사이의 불일치
- 사용자 경험을 저해하는 명백한 문제
- 서비스 간 일관성 깨짐
- PRD가 다루지 않는 엣지 케이스에서의 사용자 영향

보고 시 **사실과 영향**만 기술하고, **해결책을 선택하지 않는다**.

### 질문 효율 규칙

제품 결정 위임이 잦으면 인간의 레버리지 작업에 방해가 된다. 질문 시 다음을 지킨다:

- 같은 PR 내 여러 제품 질문이 있으면 **한 번에 모아서 질문**한다 (배치).
- 각 질문에 **선택지와 에이전트 권장안**을 포함한다 (인간의 판단 비용 최소화).
- "어떻게 할까요?"보다 "A와 B 중 어느 쪽이 맞습니까? 에이전트는 A를 권장합니다"로 질문한다.

## PRD 해석 규칙

| 상황 | 행동 |
|------|------|
| PRD에 명확히 적혀 있음 | 그대로 따른다 |
| PRD에 적혀 있으나 해석이 2개 이상 가능 | 인간에게 질문한다 |
| PRD에 적혀 있지 않음 | 구현하지 않는다 |
| PRD에 "~하면 좋겠다" 수준의 희망 사항 | 성공 기준에 포함되어 있는지 확인하고, 없으면 인간에게 확인한다 |
| PRD 간 모순 발견 | 중단하고 인간에게 보고한다 |
