# Quality Score

이 문서는 `AI-Product-Factory` 저장소의 리뷰 점수 계산 구조를 정의한다.
모든 PR은 이 문서의 기준에 따라 점수를 받고, 기준을 충족해야 병합 승인을 요청할 수 있다.

---

## 기본 원칙

- 점수는 PR 품질의 **최소 기준선**이지 완벽함의 보장이 아니다.
- 점수 통과는 **병합 자격**을 부여할 뿐, **병합 자체**를 허용하지 않는다.
- 병합의 최종 결정은 항상 **인간**이 한다.
- 모호한 경우 점수를 높게 주지 않는다 (fail-closed).

---

## 리뷰 역할

| 역할 | 관점 | 검토 항목 |
|------|------|----------|
| **critic** | 반박자 | 잘못된 가정, 빠진 엣지 케이스, 실패 시나리오, 논리적 허점 |
| **architect** | 구조 설계자 | 기존 문서/코드와의 정합성, 구조적 모순, 확장성, 범위 적절성 |
| **analyst** | 인간 운영자 대변인 | 사용자 4대 조건 반영, 운영 부담, 실용성, 명확성 |

### 각 역할의 채점 기준

#### critic (10점 만점)

| 점수 구간 | 의미 |
|-----------|------|
| 9-10 | 위험한 가정 없음, 실패 모드가 식별되고 완화됨 |
| 7-8 | 경미한 가정 위험 있으나 운영에 지장 없음 |
| 5-6 | 중요한 가정 위험 존재, 수정 필요 |
| 1-4 | 근본적 결함, 재작성 필요 |

#### architect (10점 만점)

| 점수 구간 | 의미 |
|-----------|------|
| 9-10 | 기존 문서/코드와 완전 정합, 구조적 문제 없음 |
| 7-8 | 사소한 정합성 이슈 있으나 모순 아님 |
| 5-6 | 정합성 충돌 또는 구조적 문제 존재, 수정 필요 |
| 1-4 | 기존 설계와 심각한 충돌, 재설계 필요 |

#### analyst (10점 만점)

| 점수 구간 | 의미 |
|-----------|------|
| 9-10 | 사용자 조건 완전 반영, 운영 부담 적절, 명확함 |
| 7-8 | 사용자 조건 대부분 반영, 경미한 개선 여지 |
| 5-6 | 사용자 조건 일부 미반영 또는 운영 부담 과다, 수정 필요 |
| 1-4 | 사용자 조건 무시 또는 비현실적, 재작성 필요 |

## 통과 기준

### 기준선

| 기준 | 값 | 설명 |
|------|-----|------|
| 역할별 최소 점수 | **7/10** | 어떤 역할이든 7점 미만이면 통과 불가 |
| 전체 평균 최소 점수 | **8.0/10** | 3개 역할 평균이 8.0 미만이면 통과 불가 |

### 통과 판정 로직

```
통과 = (critic ≥ 7) AND (architect ≥ 7) AND (analyst ≥ 7) AND (평균 ≥ 8.0)
```

### 판정 결과

| 결과 | 조건 | 다음 행동 |
|------|------|----------|
| **PASS** | 모든 기준 충족 | 인간에게 병합 승인 요청 |
| **REVISE** | 역할별 최소는 충족하나 평균 미달 | 가장 낮은 점수 역할의 피드백을 반영하여 수정 후 재리뷰 |
| **REJECT** | 역할별 최소 미충족 (7점 미만 존재) | 해당 역할의 피드백을 반영하여 수정 후 재리뷰 |

### 재리뷰 규칙

- 수정 후 재리뷰 시 **모든 역할**이 다시 채점한다 (부분 재채점 없음).
- 3회 연속 REJECT이면 에이전트는 중단하고 인간에게 보고한다.
- 3회 연속 REVISE (평균 미달)도 동일하게 중단하고 인간에게 보고한다.
- 즉, REJECT/REVISE 합산 3회 연속 비통과 시 자동 에스컬레이션한다.

---

## 병합 프로세스

```
에이전트 리뷰 (3개 역할)
    ↓
점수 판정 (PASS / REVISE / REJECT)
    ↓ PASS
피드백 반영 완료
    ↓
인간에게 병합 승인 요청
    ↓
인간이 최종 확인 후 merge
```

- 점수가 PASS여도 인간이 거부할 수 있다.
- 인간이 거부하면 에이전트는 거부 사유를 반영하여 수정 후 재리뷰한다.
- 인간의 merge 결정은 이 점수 체계보다 우선한다.

## 점수 기록

모든 리뷰 점수는 PR 본문 또는 커밋 메시지에 기록한다.

기록 형식:
```
멀티 에이전트 리뷰 (critic N/10, architect N/10, analyst N/10)
```

---

## PR 유형별 적용

| PR 유형 | 적용 역할 | 비고 |
|---------|----------|------|
| 문서 PR | critic, architect, analyst | 전체 적용 |
| 코드 PR | critic, architect, analyst | 전체 적용 |
| 핫픽스 | critic | architect/analyst는 사후 리뷰로 대체 가능 (인간 승인 시) |

핫픽스의 경우 인간이 "긴급"으로 판단하면 critic 단독 리뷰 + 인간 승인으로 병합할 수 있다. 단, 사후 48시간 내 전체 리뷰를 보완해야 한다. 48시간 내 보완이 이루어지지 않으면 에이전트가 인간에게 미이행을 보고하고, 해당 PR은 기술 부채로 기록한다.

## 역할 추가/변경 절차

현재는 3개 역할(critic, architect, analyst)으로 운영한다. 향후 역할 추가가 필요한 경우:

### 추가 절차

1. 새 역할의 관점, 검토 항목, 채점 기준을 정의한다.
2. 기존 역할과의 중복/공백을 분석한다.
3. 통과 기준(역할별 최소, 평균 계산)에 새 역할을 반영한다.
4. 이 문서를 갱신하고 멀티 에이전트 리뷰를 거친다.
5. 인간이 승인한다.

### 예상 추가 후보

| 역할 | 관점 | 추가 시점 |
|------|------|----------|
| verifier | 검증 완전성, 테스트 시나리오, evidence 충분성 | Sprint 1 (코드 PR 시작 시) |
| security | 보안 취약점, 인증/인가, 데이터 보호 | 외부 API/사용자 데이터 처리 시 |

역할 추가는 이 문서의 변경이므로 일반 PR과 동일한 리뷰 게이트를 거친다.

---

## Fail-Closed 기본값

- 리뷰 역할 중 하나라도 채점 불가하면 해당 PR은 REJECT 처리한다.
- 점수가 경계값(정확히 7.0 또는 8.0)이면 통과로 본다 (이상/이하가 아닌 이상).
- 점수 체계 자체에 의문이 있으면 인간에게 보고하고 수동 승인으로 전환한다.
