# Harness Common — 범용 하네스 프레임워크

이 문서는 프로젝트에 독립적인 하네스 원칙과 규칙을 담은 **범용 템플릿**이다.
새 프로젝트에 복사하여 `[프로젝트명 기입]` 항목만 채우면 즉시 적용할 수 있다.

프로젝트별 상세 규칙(서비스 라우팅, 검증 명령어, Source of Truth 경로 등)은 각 프로젝트의 CHARTER.md, AGENTS.md, WORKFLOW.md에 별도로 정의한다.

## 1. 핵심 원칙

하네스의 1차 목표는 **더 많이 생성하기**가 아니라 **잘못된 생성을 빨리 막기**다.

인간은 코드 작성에 개입하지 않는다. 인간의 역할은 시스템 설계, 스캐폴딩, 레버리지에 집중하는 것이며, 구현과 검증은 에이전트가 전담한다.

모호한 경우 허용이 아닌 차단을 기본값으로 한다 (**fail-closed**).

## 2. 소유권 모델

### 인간 소유 영역

| 항목 | 설명 |
|------|------|
| PRD 원문 작성 | 제품 요구사항의 최초 작성은 인간만 수행한다 |
| 목표 / 비목표 결정 | 무엇을 만들고 무엇을 만들지 않을지는 인간이 결정한다 |
| 성공 기준 확정 | 측정 가능한 완료 조건을 인간이 정의한다 |
| 우선순위와 범위 확정 | 작업 순서와 범위 조정은 인간 판단이다 |
| stop 조건 지정 | 어디서 멈춰야 하는지 인간이 선언한다 |
| 승인권자 지정 | spec-lock 및 최종 승인 권한은 인간에게 있다 |
| spec-lock 승인 | 실행 허가의 최종 잠금은 인간만 수행한다 |

### 에이전트 소유 영역

| 항목 | 설명 |
|------|------|
| admission 정리 보조 | 인간의 PRD를 실행 계약 형태로 정리한다 |
| 계획 분해 | ralplan을 통해 sprint/PR 단위로 분해한다 |
| 구현 | 코드, 테스트, 문서, 스크립트를 작성한다 |
| 검증 | 자동화된 검증과 evidence 수집을 수행한다 |
| 리뷰 대응 | 리뷰 피드백을 반영하고 재검증한다 |
| PR handoff | 완료된 작업을 PR로 제출한다 |

### 경계 규칙

- 에이전트는 인간 소유 항목을 **대체하지 않는다**.
- 에이전트가 인간 소유 항목에 대해 **제안**할 수는 있으나, **확정**할 수는 없다.
- 인간 소유 항목이 부재하거나 모호하면 에이전트는 **진행하지 않고 질문**한다.

## 3. 멀티 에이전트 실행 원칙

**모든 구현 작업은 멀티 에이전트로 수행한다.**

본 원칙은 실행 관련 하위 문서보다 우선한다.
하위 실행 문서(WORKFLOW 등)가 본 원칙과 충돌하면, 해당 문서를 즉시 본 원칙에 맞게 동기화한다.

| 규칙 | 설명 |
|------|------|
| 단일 에이전트 직접 구현 금지 | 에이전트가 스스로 코드를 작성하는 단일 루프 실행은 금지한다. 반드시 독립적인 서브에이전트들로 분산한다 |
| 구현 + 리뷰 동시 병렬 실행 | 구현 에이전트와 리뷰 에이전트를 동시에 띄운다. 구현이 끝나면 리뷰 결과가 이미 준비돼 있어야 한다 |
| 독립 서브에이전트 원칙 | 각 서브에이전트는 독립된 컨텍스트를 갖는다. 서브에이전트 간 결과를 공유할 때는 오케스트레이터가 명시적으로 전달한다 |

## 4. 사전 리뷰 조건 수집 프로토콜

**작업 시작 전, 인간에게 리뷰 조건을 확인한다.**

### 절차

1. 인간이 구현 요청을 하면, 에이전트는 즉시 구현에 들어가지 않는다.
2. 먼저 다음을 확인한다:
   - 이 작업에 특별히 통과해야 하는 조건이 있는가?
   - 리뷰 기준으로 추가해야 할 항목이 있는가?
3. 인간이 조건을 제시하면 → 해당 조건을 리뷰 게이트에 추가하고 작업 시작.
4. 인간이 조건 없다고 하면 → 기본 품질 점수 기준(섹션 8)으로 진행.

### 조건 미통과 시

- 제시된 조건 중 **하나라도** 통과하지 못하면 해당 작업은 **실패**로 간주한다.
- 에이전트는 실패 원인을 보고하고 작업을 **재시작**한다.
- 재시작 시 이전 실패 원인을 컨텍스트에 포함한다.

### 코드 리뷰 필수

코드 변경이 포함된 모든 작업은 병합 전 코드 리뷰를 **반드시** 수행한다. 리뷰 건너뜀은 금지한다.

## 5. 공식 실행 경로

```
PRD → admission → spec-lock → ralplan → sprint/PR 분해 → ralph 실행 → 리뷰 점수 게이트
```

| 단계 | 소유자 | 게이트 조건 |
|------|--------|------------|
| PRD | 인간 | 작성 완료 |
| admission | 에이전트 (보조) | 필수 필드 충족, 자동 반려 조건 미해당 |
| spec-lock | 인간 (승인) | 인간이 명시적으로 잠금 승인 |
| ralplan | 에이전트 | spec-lock 완료 상태 확인 |
| sprint/PR 분해 | 에이전트 | ralplan 출력 존재 |
| ralph 실행 | 에이전트 | 분해된 PR 단위 작업 수행 |
| 리뷰 점수 게이트 | 에이전트 (판정) + 인간 (최종) | 품질 점수 기준 충족 |

### 경로 이탈 금지

- 위 경로를 건너뛰거나 순서를 바꿀 수 없다.
- 어떤 단계든 게이트 조건을 충족하지 못하면 다음 단계로 넘어가지 않는다.
- 경로 변경이 필요하면 인간에게 변경 사유를 보고하고 승인을 받는다.

## 6. Stop Rules

에이전트는 다음 상황에서 **즉시 중단**하고 인간에게 보고한다.

| 규칙 | 설명 |
|------|------|
| 기획 이탈 감지 | PRD 또는 spec-lock된 요구사항과 다른 방향으로 진행 중임을 인지한 경우 |
| 모호성 해소 불가 | 요구사항 해석이 2개 이상 가능하고 에이전트 스스로 판단할 근거가 없는 경우 |
| 범위 확대 유혹 | 현재 PR의 허용 수정 범위를 넘는 변경이 필요해 보이는 경우 |
| 외부 의존성 장벽 | 외부 API, 서비스, 라이브러리의 제약으로 진행 불가한 경우 |
| 리뷰 점수 미달 | 리뷰 게이트에서 기준 점수를 충족하지 못한 경우 |
| blocker 발견 | 해결할 수 없는 기술적 장벽을 발견한 경우 |

### 중단 시 행동

1. 현재 상태를 기록한다 (무엇을 하고 있었고, 어디서 막혔는지).
2. 인간에게 보고한다 (중단 사유 + 제안 선택지).
3. 인간의 지시가 올 때까지 해당 작업을 진행하지 않는다.

---

## 7. 금지 조항

에이전트가 **절대 해서는 안 되는 것**.

| 금지 조항 | 위반 시 조치 |
|-----------|-------------|
| spec-lock 전 실행 금지 | spec-lock이 없는 상태에서 구현/코드 작성을 시작하면 즉시 중단하고 롤백한다 |
| 잠긴 요구사항 임의 수정 금지 | spec-lock된 항목을 인간 승인 없이 변경하면 해당 PR을 반려한다 |
| source-of-truth 밖 추측 기반 진행 금지 | 공식 문서에 없는 문서나 구두 합의를 근거로 진행하면 중단한다 |
| 점수 미달 상태에서 다음 단계 진행 금지 | 리뷰 게이트 미통과 PR을 다음 작업의 전제로 사용하면 중단한다 |
| blocker 은폐 금지 | 발견한 blocker를 보고하지 않고 우회하면 해당 작업 전체를 재검토한다 |

## 8. 리뷰 게이트 및 품질 점수

### 리뷰 역할

| 역할 | 관점 | 검토 항목 |
|------|------|----------|
| **critic** | 반박자 | 잘못된 가정, 빠진 엣지 케이스, 실패 시나리오, 논리적 허점 |
| **architect** | 구조 설계자 | 기존 문서/코드와의 정합성, 구조적 모순, 확장성, 범위 적절성 |
| **analyst** | 인간 운영자 대변인 | 사용자 조건 반영, 운영 부담, 실용성, 명확성 |
| **verifier** | 검증 감사자 | 테스트 완전성, 의미 있는 케이스 커버리지, evidence 충분성 |

### 채점 기준 (각 역할 10점 만점)

| 점수 | 의미 |
|------|------|
| 9-10 | 완전함. 엣지 케이스 커버, evidence 완전 |
| 7-8 | 핵심 커버, 일부 누락이나 운영에 지장 없음 |
| 5-6 | 중요한 누락 존재, 수정 필요 |
| 1-4 | 형식적이거나 근본적 결함, 재작성 필요 |

### 통과 기준

```
통과 = (critic ≥ 7) AND (architect ≥ 7) AND (analyst ≥ 7) AND (verifier ≥ 7) AND (평균 ≥ 8.0)
```

| 결과 | 조건 | 다음 행동 |
|------|------|----------|
| **PASS** | 모든 기준 충족 | 인간에게 병합 승인 요청 |
| **REVISE** | 역할별 최소는 충족, 평균 미달 | 가장 낮은 점수 역할 피드백 반영 후 재리뷰 |
| **REJECT** | 역할별 최소 미충족 (7점 미만 존재) | 해당 역할 피드백 반영 후 재리뷰 |

### 재리뷰 에스컬레이션

- 수정 후 재리뷰 시 **모든 역할**이 다시 채점한다.
- REJECT/REVISE 합산 **3회 연속** 시 즉시 중단하고 인간에게 에스컬레이션한다.

### 점수 기록 형식

```
멀티 에이전트 리뷰 (critic N/10, architect N/10, analyst N/10, verifier N/10)
```

### PR 유형별 적용

| PR 유형 | 적용 역할 | 비고 |
|---------|----------|------|
| 코드 PR | critic, architect, analyst, verifier | verifier는 테스트 완전성 + evidence 심사 |
| 문서 PR | critic, architect, analyst, verifier | verifier는 evidence record 충분성 심사 |
| 핫픽스 | critic | 나머지는 사후 48시간 내 전체 리뷰 보완 필수 |

## 9. PR 분해 기준

### PR 크기 기준

| 기준 | 권장 | 상한 | 초과 시 |
|------|------|------|---------|
| 변경 파일 수 | 1-5개 | 10개 | 분할 필수 |
| 변경 라인 수 (추가+삭제) | 50-200줄 | 400줄 | 분할 필수 |
| 변경 서비스 수 | 1개 | 1개 | cross-cutting 명시 시에만 예외 |
| 문서 전용 PR 라인 수 | — | 600줄 | 논리적 단위 분리 원칙 유지 |

자동 생성 파일(`package-lock.json` 등)은 라인 수 계산에서 제외한다.

### 분할 불가 예외 사유 (PR 본문에 명시 필수)

| 사유 | 예시 |
|------|------|
| 원자적 변경 | 타입 변경이 10개 이상 파일에 연쇄 영향 |
| 마이그레이션 | 스키마 변경 + 코드 적응이 분리 불가 |
| 초기 스캐폴딩 | 새 서비스의 최초 디렉토리/설정 구조 생성 |

### Sprint 기준

| 기준 | 권장 | 상한 |
|------|------|------|
| sprint당 PR 수 | 3-5개 | 8개 |
| sprint 기간 (wall-clock, 리뷰 대기 제외) | 1-3일 | 5일 |

### 하나의 PR에 묶지 말아야 하는 것

| 금지 조합 | 사유 |
|-----------|------|
| 리팩토링 + 기능 추가 | 롤백 시 분리 불가 |
| 여러 서비스 동시 변경 | 배포 영향 범위 확대 |
| 버그 수정 + 기능 추가 | 롤백 시 버그 수정까지 되돌려짐 |
| 의존성 업그레이드 + 코드 변경 | 문제 원인 추적 어려움 |

## 10. Spec-Lock 규칙

### 잠금 조건 (모두 충족 필요)

| 조건 | 설명 |
|------|------|
| admission PASS | admission 단계를 통과한 PRD여야 한다 |
| `[모호]` 항목 해소 | 모든 모호 태그가 인간에 의해 해소돼야 한다 |
| `[미정]` 항목 처리 | 각 항목에 대해 결정 / 에이전트 위임 명시 / scope-out 중 하나 수행 |
| 인간의 명시적 승인 | 묵시적 동의나 무응답은 승인이 아니다 |

### 잠금 후 변경 불가 항목

목표, 비목표, 성공 기준, 대상 서비스, 허용/금지 수정 범위, 불변조건, 검증 증거 요구사항.

### 에이전트 자율 조정 가능 항목 (사후 보고 필수)

PR 분해 순서, 기술 구현 방식(결과 동일), 테스트 전략 세부사항.

### 변경 필요 시

1. 에이전트가 발견 → 즉시 중단, 인간에게 보고.
2. 인간이 변경 범위 결정.
3. spec-lock 해제 → 재admission → 재spec-lock → ralplan 재실행.

## 11. 용어 사전

| 용어 | 정의 |
|------|------|
| PRD | Product Requirements Document. 인간이 작성하는 제품 요구사항 문서 |
| admission | PRD를 실행 가능한 계약으로 승격하는 검증 단계 |
| spec-lock | admission을 통과한 PRD를 인간이 잠그는 행위. 잠금 이후에만 실행 허용 |
| ralplan | 모호한 실행 요청을 계획 단계로 우회시키는 워크플로우 |
| ralph | spec-lock 이후의 반복 실행 루프 (구현→검증→리뷰→수정→재검증) |
| 리뷰 게이트 | PR 병합 전 에이전트 리뷰 + 인간 최종 승인을 받는 관문 |
| evidence trail | 검증 수행 결과를 구조화하여 기록한 증거 |
| stop rule | 에이전트가 즉시 중단해야 하는 조건 |
| harness | 에이전트의 실행을 안전하게 제약하는 시스템, 스캐폴딩, 규칙의 총체 |
| fail-closed | 판단 불가 시 허용이 아닌 차단을 기본값으로 하는 원칙 |
| critic | 리뷰 역할: 잘못된 가정과 실패 모드를 찾는 반박자 |
| architect | 리뷰 역할: 구조적 정합성과 확장성을 검토하는 설계자 |
| analyst | 리뷰 역할: 인간 운영자 관점에서 실용성을 검토하는 대변인 |
| verifier | 리뷰 역할: 테스트 완전성과 evidence를 감사하는 검증자 |

## 12. 프로젝트 적용 시 체크리스트

새 프로젝트에 이 하네스를 적용할 때 프로젝트별로 정의해야 하는 항목들.

### 필수 정의 항목

```
[ ] 프로젝트명 / 저장소 URL
[ ] 서비스 목록 및 각 서비스의 target path
[ ] 서비스별 검증 명령어 (예: cd service-a && npm run verify)
[ ] 이슈 트래커 종류 및 프로젝트 식별자 (Linear, GitHub Issues 등)
[ ] 이슈 라우팅 규칙 (라벨 → target path 매핑)
[ ] Source of Truth 문서 목록 (프로젝트별 CHARTER, AGENTS, ARCHITECTURE 등)
[ ] 브랜치 전략 (base branch명, 브랜치 네이밍 규칙)
[ ] 배포 환경 및 명령어 (staging, production)
```

### 권장 정의 항목

```
[ ] 스택별 추가 금지 조항 (예: 특정 라이브러리 사용 금지)
[ ] 프로젝트별 Stop Rule 추가 항목
[ ] 핫픽스 승인권자
[ ] 외부 리뷰 도구 연동 (CodeRabbit, Codex 등)
```
