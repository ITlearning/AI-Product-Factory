# Admission

이 문서는 PRD를 실행 가능한 계약으로 승격하는 검증 단계(admission)의 규칙을 정의한다.
admission은 PRD의 품질을 보장하고, spec-lock 전에 실행 불가능한 요청을 걸러내는 게이트다.

---

## Purpose

인간이 작성한 PRD가 에이전트 실행에 충분한 정보를 담고 있는지 검증한다.
admission을 통과한 PRD만 spec-lock 후보가 된다.

admission의 목적은 **PRD를 평가하는 것**이지 **PRD를 작성하는 것**이 아니다.
에이전트는 필드 누락을 지적하고 정리를 보조할 수 있으나, 제품 결정을 대신하지 않는다.

---

## Execution Path 상의 위치

```
PRD → [admission] → spec-lock → ralplan → sprint/PR 분해 → ralph 실행 → 리뷰 점수 게이트
```

- **입력**: 인간이 작성한 PRD 원문
- **소유자**: 에이전트 (보조) — CHARTER.md Execution Contract 참조
- **출력**: 정리된 실행 계약 (admission output)
- **다음 단계**: 인간의 spec-lock 승인

---

## 필수 필드

PRD가 admission을 통과하려면 다음 필드를 모두 포함해야 한다.

| 필드 | 설명 | 부재 시 |
|------|------|---------|
| 제목 | 프로젝트/기능의 이름 | 자동 반려 |
| 목표 | 이 프로젝트가 달성하려는 것 (1개 이상의 명확한 목표) | 자동 반려 |
| 비목표 | 이 프로젝트가 하지 않는 것 (범위 제한) | 자동 반려 |
| 성공 기준 | 측정 가능한 완료 조건 (1개 이상) | 자동 반려 |
| 대상 서비스 | 변경 대상 서비스 경로 (Repo Map 기준) | 자동 반려 |
| 사용자 시나리오 | 최소 1개의 사용자 행동 흐름 | 자동 반려 |
| 우선순위 | 작업 순서 또는 중요도 표시 | 인간에게 질문 |
| 기술 제약 | 특정 기술, 라이브러리, 아키텍처 제약 (있는 경우) | 없으면 "없음" 명시 |
| 비고 | 추가 맥락, 참고 자료, 관련 이슈 | 선택 사항 |

### 필드 기준

- **자동 반려** 필드가 하나라도 비어 있으면 admission을 통과할 수 없다.
- **인간에게 질문** 필드가 비어 있으면 에이전트가 인간에게 보완을 요청한다.
- **선택 사항** 필드는 비어 있어도 admission을 통과한다.
- 필드가 존재하되 내용이 모호한 경우, 에이전트는 해석하지 않고 인간에게 명확화를 요청한다.

---

## 자동 반려 조건

다음 조건 중 하나라도 해당하면 admission은 자동 반려된다. 에이전트는 반려 사유를 인간에게 보고한다.

| 조건 | 설명 |
|------|------|
| 필수 필드 누락 | 자동 반려 대상 필드가 하나라도 비어 있음 |
| 대상 서비스 미존재 | 대상 서비스가 Repo Map에 없고, 신규 서비스 생성 의도도 명시되지 않음 |
| 성공 기준 측정 불가 | 성공 기준에 숫자, 비율, 존재/부재, 비교 기준 중 하나 이상이 없으면 측정 불가로 판정한다. 예: "빠르다" → 불가, "200ms 이내" → 가능, "기존 대비 개선" → 불가(기준선 없음) |
| 목표와 성공 기준 불일치 | 목표에 명시된 것이 성공 기준에 반영되지 않거나 그 역 |
| 범위 과대 | 단일 PRD가 2개 이상의 서비스를 동시에 변경하려 함. PRD에 cross-cutting 사유와 대상 서비스 목록이 명시된 경우 예외. cross-cutting 여부가 불명확하면 인간에게 확인한다 |
| 기존 PRD와 충돌 | 이미 실행 중인 PRD의 범위와 겹침 |

### 자동 반려 시 행동

1. 반려 사유를 구체적으로 기술한다 (어떤 필드가, 왜 부족한지).
2. 가능하면 수정 방향을 제안한다 (단, 제품 결정을 대신하지 않는다).
3. 인간이 PRD를 보완한 후 재제출하면 처음부터 다시 검증한다.

---

## Admission 프로세스

### 1단계: 필수 필드 검증

에이전트는 PRD의 필수 필드 존재 여부를 확인한다.

- 자동 반려 대상 필드가 하나라도 누락 → 즉시 반려 + 사유 보고
- 모든 자동 반려 대상 필드 존재 → 2단계로 진행

### 2단계: 자동 반려 조건 검증

에이전트는 자동 반려 조건 테이블의 6개 조건을 모두 검사하고, 해당하는 조건을 전부 반려 사유에 포함한다.

- 하나 이상 해당 → 반려 + 모든 해당 사유 보고 (인간이 한 번에 보완할 수 있도록)
- 모두 미해당 → 3단계로 진행

### 3단계: 실행 계약 정리

에이전트는 PRD를 실행 계약 형태로 정리한다.

| 항목 | 설명 |
|------|------|
| PRD 원문 보존 | PRD 원문은 수정하지 않는다. 정리본은 별도로 작성한다 |
| 정리 범위 | 구조화, 필드 배치. 내용 변경 금지. 용어는 CHARTER.md Glossary에 정의된 것만 통일하며, 변경 시 `[용어 통일: X→Y]` 태그를 붙인다 |
| 모호성 표시 | 해석이 2개 이상 가능한 항목은 `[모호: ...]` 태그를 붙여 인간에게 질문한다 |
| 누락 추정 표시 | PRD에 명시되지 않았으나 구현에 필요할 수 있는 결정은 `[미정: ...]` 태그를 붙인다 |

### 4단계: 인간에게 제출

에이전트는 정리된 실행 계약을 인간에게 제출하며 다음을 포함한다:

- admission 통과 여부 (PASS)
- 정리된 실행 계약
- `[모호]` 및 `[미정]` 태그가 있는 경우 해당 목록과 질문
- spec-lock 요청 (인간이 잠금 승인하면 다음 단계로 진행)

## Admission Output 형식

```markdown
## Admission Output

- **PRD 제목**: [제목]
- **대상 서비스**: [서비스 경로]
- **admission 결과**: PASS / REJECT
- **반려 사유** (REJECT인 경우): [사유]

### 목표
[정리된 목표]

### 비목표
[정리된 비목표]

### 성공 기준
- [ ] [측정 가능한 기준 1]
- [ ] [측정 가능한 기준 2]

### 사용자 시나리오
[정리된 시나리오]

### 우선순위
[인간이 명시한 우선순위, 미명시 시 인간에게 질문]

### 기술 제약
[있으면 기술, 없으면 "없음"]

### 허용/금지 수정 범위
- 허용: [PRD에서 명시한 수정 허용 범위]
- 금지: [PRD에서 명시한 수정 금지 범위]

### 불변조건
[기술 제약 외 유지해야 하는 불변 조건. PRD에 명시되지 않으면 ARCHITECTURE.md Invariants 참조]

### 검증 증거 요구사항
[PRD에서 요구하는 검증 증거 유형. 명시되지 않으면 `[미정: 검증 증거 유형]`으로 표기]

### 미결 사항
- [모호: ...] — 질문
- [미정: ...] — 질문

### spec-lock 요청
인간의 승인을 요청합니다. `[모호]`/`[미정]` 항목이 있으면 spec-lock 전에 해소하거나, 에이전트에게 기술 판단 위임을 명시적으로 허가하거나, 해당 항목을 scope-out 해 주세요.
```

## Fail-Closed 기본값

- 필수 필드 존재가 불확실하면 누락으로 취급한다.
- 성공 기준의 측정 가능성이 모호하면 측정 불가로 취급한다.
- PRD 해석이 2개 이상 가능하면 에이전트가 선택하지 않고 인간에게 질문한다.
- admission과 spec-lock 사이에 PRD가 변경되면 admission을 재실행한다.
