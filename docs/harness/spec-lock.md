# Spec-Lock

이 문서는 admission을 통과한 PRD를 잠그는 규칙을 정의한다.
spec-lock은 실행 허가의 최종 관문이며, 잠금 이후에만 에이전트가 구현을 시작할 수 있다.

---

## Purpose

spec-lock은 **인간과 에이전트 사이의 실행 계약을 확정**하는 행위다.

- 잠금 전: 요구사항은 변경 가능하고, 에이전트는 구현하지 않는다.
- 잠금 후: 요구사항은 고정되고, 에이전트는 계약대로 실행한다.

spec-lock 없이 실행하는 것은 CHARTER.md Prohibited Actions의 첫 번째 금지 조항이다.

---

## Execution Path 상의 위치

```
PRD → admission → [spec-lock] → ralplan → sprint/PR 분해 → ralph 실행 → 리뷰 점수 게이트
```

- **입력**: admission output (정리된 실행 계약)
- **소유자**: 인간 (승인) — CHARTER.md Execution Contract 참조
- **출력**: 잠긴 실행 계약 (locked spec)
- **다음 단계**: ralplan (계획 분해)

---

## 잠금 조건

인간이 spec-lock을 승인하려면 다음 조건이 모두 충족되어야 한다.

| 조건 | 설명 |
|------|------|
| admission PASS | admission 단계를 통과한 PRD여야 한다 |
| `[모호]` 항목 해소 | admission output의 모든 `[모호: ...]` 태그가 인간에 의해 해소되어야 한다 |
| `[미정]` 항목 처리 | 모든 `[미정: ...]` 항목에 대해 (a) 인간이 결정, (b) 에이전트에게 기술 판단 위임을 명시적으로 허가, (c) scope-out 중 하나가 수행되어야 한다 |
| 인간의 명시적 승인 | 인간이 "spec-lock 승인" 또는 동등한 명시적 의사 표시를 해야 한다. 묵시적 동의나 무응답은 승인이 아니다 |

### 잠금 불가 상태

다음 상태에서는 spec-lock을 수행할 수 없다.

| 상태 | 사유 |
|------|------|
| admission 미통과 | 실행 경로 순서 위반 |
| `[모호]` 미해소 | 해석이 분기되는 요구사항으로 실행 불가 |
| `[미정]` 미처리 | 구현 중 Stop Rule 트리거 예상 |
| 인간 미승인 | 소유권 위반 |

---

## 잠금 효과

spec-lock이 승인되면 다음 효과가 발생한다.

| 효과 | 설명 |
|------|------|
| 요구사항 고정 | admission output의 모든 필드가 변경 불가 상태가 된다 |
| 실행 허가 | 에이전트가 ralplan → sprint/PR 분해 → ralph 실행을 시작할 수 있다 |
| 변경 절차 활성화 | 잠긴 요구사항의 변경은 아래 "잠금 해제" 절차를 따라야 한다 |
| ralplan 입력 확정 | 잠긴 목표, 비목표, 성공 기준, 대상 경로, 허용/금지 범위, 불변조건, 검증 증거 요구사항이 ralplan의 공식 입력이 된다 |

---

## 잠긴 상태에서의 변경 규칙

### 변경 불가 항목

spec-lock 이후 다음 항목은 인간 승인 없이 변경할 수 없다.

| 항목 | 사유 |
|------|------|
| 목표 | 프로젝트 방향 변경 |
| 비목표 | 범위 변경 |
| 성공 기준 | 완료 조건 변경 |
| 대상 서비스 | 영향 범위 변경 |
| 허용/금지 수정 범위 | 에이전트 행동 범위 변경 |
| 불변조건 | 시스템 안정성 기준 변경 |
| 검증 증거 요구사항 | 완료 판정 기준 변경 |

### 에이전트 자율 조정 가능 항목

다음 항목은 에이전트가 spec-lock 해제 없이 조정할 수 있다. 단, 사후 보고 필수.

| 항목 | 조건 |
|------|------|
| PR 분해 순서 | 성공 기준과 대상 서비스에 영향 없음 |
| 기술 구현 방식 | 성공 기준 충족 방법의 변경 (결과 동일) |
| 테스트 전략 세부사항 | 검증 증거 요구사항의 범위 내 |

### 에이전트 위임 판단의 관리

`[미정]` 항목에 대해 에이전트에게 기술 판단이 위임된 경우:

- 에이전트는 판단 결과를 기록하고 인간에게 사후 보고한다.
- 판단 결과가 변경 불가 항목에 영향을 미치는 경우, CHARTER.md Stop Rule "기획 이탈 감지"를 트리거하고 즉시 중단한다.

### 변경이 필요한 경우

잠긴 요구사항에 변경이 필요하면:

1. **에이전트가 발견**: 즉시 중단하고 인간에게 보고한다 (CHARTER.md Stop Rule "기획 이탈 감지" 적용).
2. **인간이 결정**: 인간이 변경 범위와 방향을 결정한다.
3. **잠금 해제**: 아래 "잠금 해제" 절차를 따른다.

## 잠금 해제

### 해제 조건

spec-lock 해제는 다음 상황에서만 수행한다.

| 상황 | 설명 |
|------|------|
| 요구사항 변경 필요 | 실행 중 목표/범위/성공 기준 변경이 불가피한 경우 |
| blocker 발견 | 현재 스펙으로는 구현 불가능한 기술적 장벽 발견 |
| 외부 환경 변화 | 외부 API 변경, 법적 요구사항 변경 등 |

### 해제 프로세스

1. **인간이 해제 승인**: spec-lock 해제는 인간만 수행할 수 있다.
2. **진행 중 작업 중단**: 해당 PRD의 모든 진행 중 PR은 중단하고 보류한다. 이미 완료된 PR은 기본적으로 유지하되, 변경된 항목에 의존하는 완료 PR은 재검증 대상으로 포함한다 (PLANS.md 변경 절차 참조).
3. **PRD 수정**: 인간이 PRD를 수정한다.
4. **재admission**: 수정된 PRD는 admission을 처음부터 다시 통과해야 한다.
5. **재spec-lock**: 인간이 다시 잠금 승인한다.
6. **ralplan 재실행**: 변경된 스펙에 맞춰 계획을 재수립한다.

### 해제 시 기록

spec-lock 해제 시 다음을 기록한다:

- 해제 사유
- 해제 시점의 진행 상태 (완료 PR, 진행 중 PR, 미착수 PR)
- 변경된 항목 목록
- 완료 PR 중 재검증이 필요한 PR 목록

기록 위치와 포맷은 [`evidence-trail.md`](evidence-trail.md)에서 정의한다.

## Fail-Closed 기본값

- 잠금 승인 여부가 모호하면 미승인으로 취급한다.
- 잠긴 요구사항의 변경 가능 여부가 모호하면 변경 불가로 취급한다.
- 에이전트 자율 조정 범위가 모호하면 인간에게 확인한다.
- 잠금 해제 없이 요구사항이 사실상 변경된 경우, 에이전트는 즉시 중단하고 보고한다.
