# Sprint/PR 분해 규칙

이 문서는 ralplan 출력을 sprint와 PR 단위로 분해할 때의 수치 기준과 세부 규칙을 정의한다.
PLANS.md의 Sprint/PR 분해 원칙을 구체화하며, 원칙과 충돌 시 PLANS.md가 우선한다.

---

## Purpose

ralplan이 생성한 계획을 실행 가능한 sprint/PR 단위로 분해하는 기준을 제공한다.
분해의 목표는 **리뷰 가능성, 롤백 가능성, 검증 가능성**을 보장하는 것이다.

---

## Execution Path 상의 위치

```
PRD → admission → spec-lock → ralplan → [sprint/PR 분해] → ralph 실행 → 리뷰 점수 게이트
```

- **입력**: ralplan 출력 (sprint 목록, PR 목록, 의존성 그래프)
- **소유자**: 에이전트 — CHARTER.md Execution Contract 참조
- **출력**: 실행 가능한 sprint/PR 단위 계획
- **다음 단계**: ralph 실행 (구현→검증→리뷰→수정→재검증 루프)

## PR 크기 기준

### 수치 기준

| 기준 | 권장 | 상한 | 초과 시 |
|------|------|------|---------|
| 변경 파일 수 | 1-5개 | 10개 | 분할 필수 |
| 변경 라인 수 (추가+삭제) | 50-200줄 | 400줄 | 분할 필수 |
| 변경 서비스 수 | 1개 | 1개 | cross-cutting 명시 시에만 예외 (admission.md 참조, ARCHITECTURE.md Invariant #6) |

### 기준 적용 규칙

- **상한 초과**: 에이전트는 PR을 분할해야 한다. 분할 불가능한 경우 사유를 기록하고 인간에게 보고한다.
- **권장 범위 내**: 추가 조치 없이 진행한다.
- **문서 전용 PR**: 변경 라인 수 상한을 600줄로 완화한다 (문서는 코드 대비 리뷰 인지 부하가 낮고 자동 검증이 적으므로). 단, 논리적 단위 분리 원칙은 유지한다.
- **자동 생성 파일**: `package-lock.json` 등 자동 생성 파일의 변경은 라인 수 계산에서 제외한다.

### 예외 사유 (분할 불가능한 경우)

다음 사유가 있으면 상한 초과를 허용하되, PR 본문에 명시해야 한다:

| 사유 | 예시 |
|------|------|
| 원자적 변경 | 타입 변경이 10개 이상의 파일에 연쇄 영향 |
| 마이그레이션 | 스키마 변경 + 코드 적응이 분리 불가 |
| 초기 스캐폴딩 | 새 서비스의 최초 디렉토리/설정 구조 생성 |

## Sprint 분해 기준

### 수치 기준

| 기준 | 권장 | 상한 | 초과 시 |
|------|------|------|---------|
| sprint당 PR 수 | 3-5개 | 8개 | sprint 분할 필수 |
| sprint 기간 (에이전트 추정, wall-clock 기준, 리뷰 대기 제외) | 1-3일 | 5일 | sprint 분할 필수 |

### Sprint 구성 규칙

| 규칙 | 설명 |
|------|------|
| 단일 목표 | 하나의 sprint는 하나의 명확한 목표를 가진다 (PLANS.md 원칙) |
| 측정 가능한 완료 조건 | sprint 완료 시 독립적으로 검증 가능한 결과물이 있어야 하며, 완료 조건이 측정 가능한 형태로 기술되어야 한다 (PLANS.md 원칙) |
| 의존성 최소화 | sprint 간 의존성을 최소화하고, 있으면 명시한다 |
| 순차 실행 | sprint는 번호 순서대로 실행한다. 병렬 실행하지 않는다 |

## PR 의존성 규칙

### 의존성 표현

PR 간 의존성은 ralplan 출력에 명시한다.

```
PR-1 → PR-2 → PR-3  (순차)
PR-4, PR-5           (병렬 가능)
PR-6 → PR-7          (PR-6 완료 후 PR-7 시작)
```

### 의존성 규칙

| 규칙 | 설명 |
|------|------|
| 순환 금지 | PR 의존성 그래프에 순환이 있으면 안 된다 |
| 최소 의존성 | 가능하면 PR 간 의존성을 줄여 병렬 실행 가능성을 높인다 |
| 의존성 명시 | 모든 PR 간 의존성은 계획에 명시한다. 암묵적 의존성을 허용하지 않는다 |
| 선행 PR 완료 필수 | 의존하는 PR이 리뷰 게이트를 통과(QUALITY_SCORE.md의 PASS 판정)하기 전에 후행 PR을 시작하지 않는다 |

### 의존성 위반 시

의존성이 깨진 경우 (선행 PR이 REJECT/REVISE된 경우):

1. 후행 PR 작업을 즉시 중단한다.
2. 선행 PR이 리뷰 게이트를 통과할 때까지 대기한다.
3. 선행 PR의 변경이 후행 PR에 영향을 주면 후행 PR도 재계획한다.

## PR 구성 원칙

### 하나의 PR에 포함해야 하는 것

| 항목 | 설명 |
|------|------|
| 코드 변경 | 하나의 논리적 변경에 해당하는 코드 |
| 관련 테스트 | 해당 변경을 검증하는 테스트 |
| 검증 방법 | PR 본문에 검증 방법 명시 (PLANS.md 원칙) |
| 문서 갱신 | 변경에 따른 문서 업데이트 (있는 경우) |

### 하나의 PR에 묶지 말아야 하는 것

| 금지 | 사유 |
|------|------|
| 리팩토링 + 기능 추가 | 리뷰 복잡도 증가, 롤백 시 기능과 구조 분리 불가 |
| 여러 서비스 변경 | 배포 영향 범위 확대 (ARCHITECTURE.md Invariant) |
| 버그 수정 + 기능 추가 | 롤백 시 버그 수정까지 되돌려짐 |
| 의존성 업그레이드 + 코드 변경 | 문제 원인 추적 어려움 |

## 분해 검증

에이전트는 분해 완료 후 다음을 확인한다.

| 검증 항목 | 기준 |
|-----------|------|
| PR 크기 | 모든 PR이 상한 이내 (예외 사유 있는 경우 명시) |
| PR 단일 변경 | 각 PR이 하나의 논리적 변경만 포함 |
| 검증 방법 | 모든 PR에 검증 방법이 명시됨 |
| 의존성 무순환 | 의존성 그래프에 순환 없음 |
| Sprint 목표 | 각 sprint에 명확한 단일 목표 있음 |
| 성공 기준 커버리지 | spec-lock된 모든 성공 기준이 최소 1개 PR에 매핑됨 |

검증 항목 중 하나라도 미충족이면 분해를 수정한다. 수정 불가한 경우 인간에게 보고한다.

---

## Fail-Closed 기본값

- PR 크기가 상한에 근접하면 분할을 우선한다.
- 의존성 관계가 불명확하면 순차로 처리한다 (병렬 금지).
- 분해가 적절한지 판단 불가하면 인간에게 보고한다.
- 성공 기준이 어떤 PR에도 매핑되지 않으면 분해 미완료로 취급한다.
