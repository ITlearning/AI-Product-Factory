# IBAD 첫 거절문 추천 설계안

## 작업 가정

이 문서는 별도 인간 피드백이 들어오기 전까지
`프롬프트만 다듬는 방향`이 아니라
`답장을 못 보내는 순간에 첫 거절문을 더 쉽게 고르게 해주는 방향`을 추천안으로 둔다.

이유는 현재 `IBAD/app`이 이미 최소 입력 기반의 `reply-start` 흐름까지는 줄어든 상태이기 때문이다.
지금의 어색함은 입력 개수가 많아서라기보다
`왜 이 답장을 믿고 보내도 되는지`가 약하다는 데 더 가깝다.

최근 논의 반영:

- `boundary coach`라는 표현은 1차 제품 기준으로는 다소 크게 들린다.
- 따라서 제품 전면 포지셔닝은 `코치`보다 `첫 거절문 추천` 쪽으로 낮춰 읽는다.
- 핵심 변화는 상담형 확장이 아니라, 질문 1개와 추천 우선순위 1개를 더하는 수준으로 제한한다.

## 1. 현재 상태 진단

현재 제품은 다음 구조다.

- 받은 메시지 입력
- `약속` / `부탁` 선택
- 답장 3개 생성

이 구조는 진입 장벽을 낮추는 데는 성공했지만,
아래 이유로 아직 `거절을 못하는 사람을 돕는 도구`라기보다
`문장 3개를 보여주는 생성기`처럼 느껴진다.

- 세 카드가 모두 비슷한 수준의 임의 제안처럼 보인다.
- 사용자는 `어떤 톤이 지금 나에게 맞는지`보다 `뭘 골라도 이상하면 어떡하지`를 먼저 걱정한다.
- 결과의 품질 평가는 문장 자체보다 `이걸 보내도 괜찮다`는 확신에서 갈린다.
- 현재의 `whyItWorks`는 결과 뒤에 붙는 설명일 뿐, 사용자의 멈춤 상태를 풀어주지는 못한다.

즉, 문제는 단순한 복사 품질보다
`답장을 시작할 수 있게 만드는 심리적 보조 장치가 없다`는 쪽에 더 가깝다.

## 2. 비교한 세 가지 방향

### 방향 A. 프롬프트만 개선하는 고급 거절문 생성기

- 장점: 가장 작은 구현 범위
- 장점: 기존 UI와 계약을 거의 유지할 수 있음
- 단점: 사용자가 왜 못 보내는지 자체는 여전히 다루지 못함
- 단점: 품질 인식이 모델 편차에 계속 끌려다님

이 방향은 필요하지만, 단독으로는 부족하다.

### 방향 B. 막힘 이유를 반영하는 첫 거절문 추천

- 장점: 현재의 가벼운 진입 경험을 유지하면서도 서비스 이유와 더 잘 맞음
- 장점: `왜 못 보내는지`를 한 단계만 더 받아 추천 근거를 만들 수 있음
- 장점: AI가 모든 걸 결정하지 않고, 추천 구조 일부를 제품 로직으로 고정할 수 있음
- 단점: 입력 필드가 1개 늘어나고 결과 구조가 조금 더 복잡해짐

이 방향을 추천한다.

### 방향 C. 후속 답장까지 돕는 다회차 보조 도구

- 장점: 실사용 문제를 더 깊게 다룸
- 장점: 반복 연락, 붙잡힘, 관계 정리 같은 고통 지점까지 확장 가능
- 단점: 안전 범위와 테스트 범위가 크게 넓어짐
- 단점: 현재 MVP의 좁은 약속과 충돌할 가능성이 큼

이 방향은 차기 확장으로 미루는 편이 맞다.

## 3. 추천 포지셔닝

한 줄 정의:

`IBAD`는 거절 문장을 대신 써주는 생성기가 아니라, 답장을 못 보내고 멈춘 사람에게 지금 보낼 첫 거절문을 더 쉽게 고르게 해주는 도구다.

핵심 약속:

- 완벽한 개인 맞춤 문장보다 `보내기 쉬운 시작점`을 준다.
- 사용자가 멈춘 이유를 가볍게 반영한다.
- 답장을 짧고 분명하게 만들되, 차갑게 보일까 하는 불안을 함께 낮춘다.

피해야 할 포지셔닝:

- 모든 관계와 모든 감정 문제를 다루는 상담 앱
- 후속 대화까지 책임지는 만능 비서
- 톤 커스터마이징이 많은 문장 실험실

## 4. 제안하는 MVP 변화

### 4.1 입력

첫 화면은 여전히 가볍게 유지한다.
다만 한 가지 질문을 추가한다.

- 받은 메시지
- 요청 종류: `약속` / `부탁`
- 지금 막히는 이유:
  - `미안해서 시작이 안 돼요`
  - `너무 차갑게 보일까 걱정돼요`
  - `말이 길어질까 봐 걱정돼요`

핵심은 관계 세분화가 아니라 `사용자가 멈춘 이유`를 잡는 것이다.

### 4.2 결과

결과는 `3개 카드 나열`에서 `추천된 시작점 + 대안` 구조로 바꾼다.

- 추천 카드 1개
- 대안 카드 2개
- 짧은 가이드 메모 1개
- 피해야 할 표현 1개

예시 구조:

- 추천 시작 문장: 지금 막힌 이유에 가장 맞는 카드
- 다른 선택지:
  - 조금 더 부드럽게
  - 조금 더 짧게
- 가이드 메모:
  - `설명을 길게 붙이면 다시 붙잡힐 수 있어요. 결론 한 번, 감사 한 번이면 충분해요.`
- 피해야 할 표현:
  - `나중에 보자`

이렇게 하면 사용자는 `셋 중 하나를 고른다`가 아니라
`지금 내 상황에서 먼저 믿고 보낼 카드 하나를 받는다`는 느낌을 받는다.

## 5. 제품 로직과 AI 책임 분리

이번 변경의 핵심은 AI에게 모든 판단을 맡기지 않는 것이다.

### 제품 로직이 맡을 것

- `blockerType` 옵션 정의
- `blockerType -> recommendedTone` 매핑
- 추천 카드 강조 순서
- 막힘 유형별 가이드 메모 기본 카피
- 막힘 유형별 재생성 버튼 라벨 또는 설명

### AI가 맡을 것

- 주어진 상황에 맞는 세 개의 짧은 답장 작성
- 각 답장의 짧은 이유 설명
- 피해야 할 표현 후보 1개 생성

이 분리는 품질 문제를 줄인다.
사용자가 체감하는 핵심 추천 구조를 모델 출력에 전부 의존하지 않기 때문이다.

## 6. UX 흐름

1. 사용자가 받은 메시지를 붙여넣는다.
2. `약속` 또는 `부탁`을 고른다.
3. 지금 못 보내는 이유를 고른다.
4. 서비스는 막힘 이유에 맞는 추천 카드를 먼저 보여준다.
5. 사용자는 추천 카드를 그대로 복사하거나, 옆의 대안을 고른다.
6. 필요하면 같은 막힘 이유 기준으로 다시 생성한다.

성공 기준은 `문장이 화려하다`가 아니라
`이제 하나는 보낼 수 있겠다`는 감각을 주는 것이다.

## 7. 출력 원칙

- 세 답장은 길이와 강도가 분명히 달라야 한다.
- 추천 카드는 막힘 이유와 가장 잘 맞는 톤이어야 한다.
- 가이드 메모는 심리적 마찰을 줄이는 짧은 편집 메모여야 한다.
- 결과는 분석 리포트처럼 길어지면 안 된다.
- `whyItWorks`는 추상적 칭찬 대신 보내는 데 도움이 되는 이유여야 한다.

예:

- 나쁨: `자연스럽습니다`
- 좋음: `결론을 먼저 말해 다시 잡힐 여지를 줄여요`

## 8. 범위와 비범위

이번 변경에 포함:

- 막힘 이유 선택 1단계 추가
- 추천 카드 강조
- 가이드 메모 / 피해야 할 표현 추가
- 추천 구조에 맞춘 프롬프트와 계약 변경

이번 변경에서 제외:

- 연인/가족/직장 확장
- 후속 답장 자동 생성
- 장기 대화 기억
- 사용자의 실제 말투 학습

## 9. 성공 지표

정량 후보:

- 첫 생성 후 복사 클릭률
- 첫 생성 후 이탈률 대비 재생성률
- 막힘 이유별 추천 카드 복사율

정성 후보:

- `문장이 마음에 든다`보다 `이건 보낼 수 있겠다` 반응 증가
- `무슨 톤을 골라야 할지 모르겠다` 피드백 감소

## 10. 구현 영향 메모

현재 `IBAD/app`은 이미 최소 입력 기반의 `reply-start` 앱으로 정리되어 있다.
따라서 다음 단계는 다시 많은 옵션을 넣는 것이 아니라,
현재 구조 위에 `멈춤 이유를 반영하는 얇은 추천 레이어`를 얹는 것이다.

구현 관점에서 필요한 축은 네 가지다.

- 입력 계약에 `blockerType` 추가
- 결과 계약에 `recommendedTone`, `coachNote`, `avoidPhrase` 추가
- UI에서 추천 카드를 먼저 보이도록 재구성
- 프롬프트/검증을 `추천 가능한 세트` 기준으로 강화

이 문서를 기준으로 구현 계획을 별도 작성한다.
