# Harness Engineering Claude Code Brief

## 목적

이 문서는 `AI-Product-Factory` 저장소에 하네스 엔지니어링을 도입하기 위해 지금까지 합의한 내용을 Claude Code와 공유하기 위한 브리프다. 목표는 원 논의를 다시 처음부터 반복하지 않고, 이미 잠긴 판단과 아직 열려 있는 쟁점을 분리해서 이어서 검토할 수 있게 만드는 것이다.

현재 작업 브랜치:

- `chore/harness-engineering-foundation`

---

## 원래 문제 정의

이 저장소의 원래 지향점은 다음에 가깝다.

- 인간이 프로젝트의 기획을 제공한다.
- 하네스와 멀티 에이전트가 그 기획을 바탕으로 프로젝트를 분해한다.
- 에이전트가 구현, 검증, 리뷰 대응, PR handoff를 수행한다.
- 인간은 코드 작성보다는 시스템, 스캐폴딩, 레버리지, 우선순위, 승인, 중단 판단에 집중한다.

하지만 논의 중 다음 점이 명확해졌다.

- `기획서만 던져주면 된다`는 문장은 운영 계약으로 쓰기에는 너무 강하다.
- PRD/기획은 인간이 강하게 개입하고 잠가야 한다.
- 에이전트는 PRD 이후의 실행 루프를 맡는 것이 맞다.
- 하네스의 첫 목표는 `더 많이 생성`이 아니라 `잘못된 생성을 빨리 막기`다.

## 핵심 결론

### 1. 리포 공통 하네스를 먼저 깐다

초기 적용 범위는 특정 서비스 구현이 아니라 저장소 공통 하네스다.

- 짧은 `AGENTS.md`
- 구조화된 `docs/`
- `ARCHITECTURE.md`
- `PLANS.md`
- `QUALITY_SCORE.md`
- `RELIABILITY.md`
- `SECURITY.md`
- PRD admission / spec-lock / review gate / evidence trail

### 2. PRD는 인간 소유다

다음은 인간 소유 영역으로 잠갔다.

- PRD 원문 작성
- 목표/비목표 결정
- 성공 기준 확정
- 우선순위와 범위 확정
- stop 조건과 승인권자 지정
- spec-lock 승인

에이전트는 PRD를 대체하지 않는다.

### 3. 에이전트는 PRD 이후를 맡는다

다음은 에이전트 소유 영역으로 본다.

- PRD를 실행 가능한 계약으로 정리
- 스프린트와 PR 단위로 분해
- 코드/테스트/문서/스크립트 작성
- 검증 수행
- 리뷰 대응
- PR handoff
- 나중에는 문서 가드닝, observability 기반 검증 확장

### 4. 공식 기본 경로

기본 실행 경로는 아래로 잠갔다.

`PRD -> admission -> spec-lock -> ralplan -> sprint/PR 분해 -> ralph 실행 -> 리뷰 점수 게이트`

중요:

- `ralph`는 포함된다.
- 하지만 `원본 PRD`에 바로 붙지 않는다.
- `spec-lock` 이후의 실행 루프에 붙는다.

### 5. 설계 철학

초기 철학은 다음으로 잠갔다.

- 처음부터 완벽하게 만들지 않는다.
- 작은 하네스를 먼저 깐다.
- 실패 지점이 드러나면 그것을 다음 하네스 개선 입력으로 삼는다.
- `잘 만드는 것`보다 먼저 `잘 멈추는 것`을 설계한다.

## 멀티 에이전트 검토 결과

이번 설계는 단일 의견으로 밀지 않고 여러 역할로 비평했다.

사용한 역할:

- `critic`: 실패 모드 반박
- `architect`: 운영/제약/경계 검토
- `analyst`: 인간 운영자/요구사항 관점 검토
- `writer`: 문체/가독성 검토

수렴된 결론:

- 이 저장소는 하네스 엔지니어링에 잘 맞는다.
- 다만 `기획서만 던져주면 된다`는 표현은 과장이다.
- 운영 계약은 `잠긴 PRD + 구조화된 입력 계약`이 되어야 한다.
- Sprint 0의 목표는 `잘 생성하기`가 아니라 `잘 멈추기`다.
- review gate와 evidence trail이 없으면 멀티 에이전트는 빨라질 뿐 일관되게 틀릴 수 있다.

## 사용자가 중요하다고 표시한 원칙

사용자가 OpenAI 글과 관련 자료를 읽으며 중요하다고 판단한 요지는 아래처럼 정리할 수 있다.

### 인간이 해야 하는 생각

- `어떤 기능이 누락되어 있는가`
- `에이전트가 읽기 쉽고 실행 가능하게 만들려면 무엇이 더 필요한가`
- `어디서 멈춰야 하는가`

### 에이전트가 직접 해야 하는 것

- 표준 개발 도구 직접 사용
- 컨텍스트 수집
- 리뷰 요청
- 리뷰 피드백 대응
- PR 생성 및 업데이트
- 검증 루프 반복

### Ralph Wiggum Loop 관련 해석

사용자가 강조한 문장은 사실상 이 리포에도 그대로 들어온다.

- Codex가 로컬 변경을 검토한다.
- 추가 에이전트 리뷰를 요청한다.
- 사람/에이전트 피드백에 응답한다.
- 모든 검토자가 만족할 때까지 반복한다.

즉, 하네스는 단순 구현 자동화가 아니라 `반복 검토 + 반복 수정 + 반복 검증`을 시스템에 인코딩하는 구조다.

### AGENTS.md에 대한 결론

큰 단일 `AGENTS.md`는 실패하기 쉽다는 점에 동의했다.

- 컨텍스트 낭비
- 오래된 규칙 누적
- 무엇이 중요한지 흐려짐
- 기계적 검증 어려움

따라서 이 리포도 다음 원칙을 따른다.

- `AGENTS.md`는 백과사전이 아니라 맵
- 실제 기록 시스템은 구조화된 `docs/`
- 계획은 일급 아티팩트
- 점진적 공개가 가능해야 함

## 지금까지 잠긴 설계 결정

### 운영 슬로건 수정

기존 생각:

- `기획서만 던져주면 하네스 & 멀티에이전트로 프로젝트를 만든다`

수정된 운영 문장:

- `인간이 잠근 PRD와 구조화된 입력 계약이 주어지면, 하네스와 멀티 에이전트가 프로젝트를 안전하게 분해·실행·검증·PR handoff 한다`

### 인간 vs 에이전트 책임 경계

인간:

- PRD 작성
- 제품 판단
- 우선순위
- 중단 승인
- spec-lock 승인

에이전트:

- admission 정리 보조
- 계획 분해
- 구현
- 검증
- 리뷰 대응
- PR handoff

### Ralph Loop의 위치

- `ralph`는 사용한다.
- `admission/spec-lock` 이전에는 사용하지 않는다.
- 모호한 요청은 `ralplan`으로 먼저 보낸다.

### 하네스의 1차 목표

- 더 많이 생성하기
- 완전 자동 보장

위 둘이 아니라 아래로 본다.

- 잘못된 생성을 일찍 막기
- 애매하면 자동 진행보다 중단
- 검증과 리뷰 기록으로만 진행 정당화

## PR 0-9 상세 결정

`PR 0-9`는 `좋은 PRD 양식`이 아니라 `실행 허가/반려 계약`이다.

### 목적

- 인간이 쓴 자유서술 PRD를 `spec-lock 가능한 실행 계약`으로 승격
- 모호성, 경로, 검증 가능성, stop 질문을 드러냄
- admission 통과한 것만 downstream으로 넘김

### 필수 필드

- `Project / Problem / Goal`
- `Non-Goals`
- `Success Criteria`
- `Primary Target Path`
- `Allowed Touch Surface`
- `Disallowed Areas`
- `Constraints / Invariants`
- `Dependencies / External Systems`
- `Open Questions`
- `Must Ask Before Execution`
- `Acceptance Evidence`
- `Owner / Approver`

### 인간이 직접 잠가야 하는 항목

- 목표
- 비목표
- 성공 기준
- 주요 수정 경로
- 금지 영역
- 제약/불변조건
- 실행 전 필수 질의
- 승인권자

### 에이전트가 보조 정리 가능한 항목

- 허용 수정 범위 정리
- 외부 의존성 정리
- 검증 증거 후보 정리
- 리뷰 요구사항 정리

단, `spec-lock` 전에는 인간 승인 없이는 확정되지 않는다.

### admission 통과 게이트

- 목표가 한 문장으로 설명된다
- 비목표가 존재한다
- 성공 기준이 측정 가능하다
- 대상 경로가 잠겨 있다
- 허용/금지 수정 범위가 있다
- 오픈 질문이 숨겨져 있지 않다
- 실행 전 필수 질의가 명시돼 있다
- 승인권자가 있다
- 제약/위험이 비어 있지 않다
- 통과 증거가 정의돼 있다

### 자동 반려 조건

- 목표 없이 해결책만 있음
- 성공 기준이 `좋아 보이면` 수준임
- 대상 경로가 없음
- 모순되는 요구가 있는데 우선순위가 없음
- 질문이 남아 있는데도 실행 허용으로 표시됨
- 승인권자나 source of truth가 불명확함
- 위험한 외부 의존성이 있는데 제약 설명이 없음

### downstream에 넘길 잠긴 출력

- 잠긴 목표
- 잠긴 비목표
- 잠긴 성공 기준
- 잠긴 대상 경로
- 허용/금지 수정 범위
- 오픈 질문 목록
- 실행 전 필수 질의 목록
- 검증 증거 요구사항

한 줄 요약:

- `PR 0-9는 PRD를 풍부하게 만드는 문서가 아니라, 실행을 허용할 만큼만 명확하게 만드는 문서다`

## Ralph / Ralplan 관련 최종 판단

질문:

- `랄프 루프 방식도 포함되어있는지 확인`

결론:

- `ralph` 자체는 이미 존재하는 워크플로우다.
- `ralplan`도 존재하며 모호한 실행 요청을 planning으로 우회시키는 역할을 한다.
- 따라서 `랄프 루프`는 제외 대상이 아니라 공식 실행 경로의 일부다.
- 다만 위치는 `PRD 이전`이 아니라 `spec-lock 이후`다.

잠긴 공식 경로:

`PRD -> admission -> spec-lock -> ralplan -> sprint/PR 분해 -> ralph 실행 -> 리뷰 점수 게이트`

4. `레포지토리 공통 하네스는 좋아. 그리고 이 프로젝트의 원래 취지 자체는 "기획서만 던져주면 하네스 & 멀티에이전트로 프로젝트를 만든다." 거든 그래서 하네스 엔지니어링에 가장 잘 부합한 레포지토리라고 생각했어`

6. `랄프루프 방식도 포함되어있는지 확인해줘`
7. 사용자가 하네스 엔지니어링 관련 글에서 중요하다고 본 내용을 긴 메모 형태로 붙여 넣음
