# Process Common — 범용 하네스 프레임워크

이 문서는 프로젝트에 독립적인 기획→실행 프로세스 원칙을 담은 **범용 템플릿**이다.
새 프로젝트에 복사해서 `[프로젝트명 기입]` 항목만 채우면 곧바로 적용할 수 있다.

프로젝트별 상세(서비스 라우팅, 검증 명령어, Source of Truth 경로, gstack 스킬 매핑 등)는 각 프로젝트의 `CHARTER.md`, `AGENTS.md`, `SEQUENCE.md`에 별도로 정의한다.

## 1. 핵심 원칙

[프로젝트명 기입]의 하네스 1차 목표는 **더 많이 생성하기**가 아니라 **잘못된 생성을 빨리 막기**다.

인간(소유자)은 코드 작성에 직접 개입하지 않는다. 인간의 역할은 시스템 설계, 스캐폴딩, 레버리지에 집중하는 것이며, 구현과 검증은 에이전트가 전담한다.

판단이 모호한 경우, 기본값은 진행이 아니라 **확인 질문**이다. 무리한 추측으로 진행하지 않고 인간에게 1회 묻고 답을 받는 흐름을 표준으로 둔다.

## 2. 소유권 모델

### 인간 소유 영역

| 항목 | 설명 |
|------|------|
| PRD 원문 작성 | 제품 요구사항의 최초 작성은 인간만 수행한다 |
| 목표 / 비목표 결정 | 무엇을 만들고 무엇을 만들지 않을지는 인간이 결정한다 |
| 성공 기준 확정 | 측정 가능한 완료 조건을 인간이 정의한다 |
| 우선순위와 범위 확정 | 작업 순서와 범위 조정은 인간 판단이다 |
| stop 조건 지정 | 어디서 멈춰야 하는지 인간이 선언한다 |
| 단계 승인 | 각 단계 게이트의 OK / 수정 / 중단 결정은 인간 권한이다 |

### 에이전트 소유 영역

| 항목 | 설명 |
|------|------|
| PRD 정리 보조 | 인간의 brief를 13필드 PRD 양식으로 정리한다 (시각화 2필드 — Wireframe / As-Is → To-Be 포함) |
| 계획 분해 | sprint/PR 단위 분해와 의존성 DAG를 산출한다 |
| 구현 | 코드, 테스트, 문서, 스크립트를 작성한다 |
| 검증 | 자동화된 검증과 evidence 수집을 수행한다 |
| 리뷰 대응 | 리뷰 피드백을 반영하고 재검증한다 |
| PR handoff | 완료된 작업을 PR로 제출한다 |

### 경계 규칙

- 에이전트는 인간 소유 항목을 **대체하지 않는다**.
- 에이전트가 인간 소유 항목에 대해 **제안**할 수는 있으나, **확정**할 수는 없다.
- 인간 소유 항목이 부재하거나 모호하면 에이전트는 **진행하지 않고 질문**한다.

## 3. 멀티 에이전트 실행 원칙

**독립적인 N개 작업은 멀티 에이전트로 병렬 수행한다.**

본 원칙은 실행 관련 하위 문서보다 우선한다. 하위 문서가 본 원칙과 충돌하면 즉시 본 원칙에 맞게 동기화한다.

| 규칙 | 설명 |
|------|------|
| 단일 루프 직접 구현 지양 | 독립 작업이 N개 이상이면 단일 에이전트가 순차 처리하지 않는다. 독립 서브에이전트로 분산한다 |
| 구현 + 리뷰 동시 병렬 | 구현 에이전트와 리뷰 에이전트를 함께 띄운다. 구현이 끝나는 시점에 리뷰 결과가 준비돼 있어야 한다 |
| 독립 컨텍스트 원칙 | 각 서브에이전트는 독립된 컨텍스트를 갖는다. 결과 공유는 오케스트레이터가 명시적으로 수행한다 |
| 단일 메시지 dispatch | 같은 라운드 병렬 에이전트는 한 메시지에 묶어 동시에 띄운다 |

병렬은 **[4] PARALLEL REVIEW**와 **[6] EXECUTION** 두 단계에서만 활성화한다. 그 외 단계는 1:1 대화 흐름이다 (5장 참고).

## 4. 사전 리뷰 조건 수집 프로토콜

**작업 시작 전, 인간에게 리뷰 조건을 확인한다.**

### 절차

1. 인간이 구현 요청을 하면, 에이전트는 즉시 구현에 들어가지 않는다.
2. 먼저 다음을 확인한다:
   - 이 작업에 특별히 통과해야 하는 조건이 있는가?
   - 리뷰 기준으로 추가해야 할 항목이 있는가?
3. 인간이 조건을 제시 → 해당 조건을 리뷰 게이트에 추가하고 작업 시작.
4. 인간이 조건 없다고 응답 → 8장의 기본 리뷰 골격으로 진행.

### 조건 미통과 시

- 제시된 조건 중 **하나라도** 통과하지 못하면 해당 작업은 **실패**로 간주한다.
- 에이전트는 실패 원인을 보고하고 작업을 **재시작**한다.
- 재시작 시 이전 실패 원인을 컨텍스트에 포함한다.

### 코드 리뷰 필수

코드 변경이 포함된 모든 작업은 병합 전 코드 리뷰를 **반드시** 수행한다. 리뷰 건너뜀은 금지한다.

## 5. 공식 실행 경로

[프로젝트명 기입]은 다음 6단계 시퀀스를 따른다.

```
[1] INTAKE → [2] DISCOVERY → [3] PRD DRAFT → [4] PARALLEL REVIEW → [5] DECOMPOSITION → [6] EXECUTION
```

### 단계 요약

| 단계 | 소유자 | 출력 | 인간 승인 패턴 |
|------|--------|------|--------------|
| [1] INTAKE | 자동 분기 | [2] 또는 [3]으로 라우팅 결정 | 분기 신호 상충 시에만 1회 확인 |
| [2] DISCOVERY | 1 에이전트 + 인간 | 1쪽 brief | full 대화: "이 brief로 [3] 갈까?" |
| [3] PRD DRAFT | 1 에이전트 + 인간 | 13필드 PRD (Wireframe / As-Is → To-Be 포함) | full 대화: "이 PRD로 [4] 갈까?" |
| [4] PARALLEL REVIEW | 3 에이전트 병렬 + 인간 | 3개 리뷰 보고서 + 합의 액션 | full 대화: "이 결과 반영해 [5] 갈까?" |
| [5] DECOMPOSITION | 1 에이전트 + 인간 | Sprint 표 + PR 표 (DAG) | full 대화: "이 분해로 [6] 갈까?" |
| [6] EXECUTION | N 에이전트 병렬 | 머지된 PR + evidence trail | PR 단위로 인간이 결과 확인 |

### 흐름 규칙

- **단계마다 인간 승인 필수.** [2]→[3], [3]→[4], [4]→[5], [5]→[6] 모든 전이는 게이트 통과 후 진입.
- **자동 진행 안 함.** 이전 산출물이 완성되어도 다음 단계로 자동 점프하지 않는다. 표준 질문: "이 결과로 [다음 단계 이름] 갈까?"
- **인간 답변은 셋 중 하나:** OK / 수정 요청 / 중단.
- **산출물 저장 우선.** 게이트 질문 전에 산출물을 `docs/plans/` 또는 `docs/reviews/`(또는 [프로젝트명 기입]의 해당 경로)에 먼저 쓴다.
- **후퇴 가능.** [4]에서 PRD 수정이 필요하면 [3]으로 되돌려 갱신 후 [4] 재진입.
- **병렬은 [4]와 [6]에서만.** 나머지 단계는 1:1 대화 흐름이다.

### 경로 이탈 금지

- 위 경로를 건너뛰거나 순서를 바꿀 수 없다.
- 게이트 조건을 충족하지 못하면 다음 단계로 넘어가지 않는다.
- 경로 변경이 필요하면 인간에게 변경 사유를 보고하고 승인을 받는다.

## 6. Stop Rules

에이전트는 다음 상황에서 **즉시 중단**하고 인간에게 보고한다.

| 규칙 | 설명 |
|------|------|
| 기획 이탈 감지 | PRD 또는 합의된 요구사항과 다른 방향으로 진행 중임을 인지한 경우 |
| 모호성 해소 불가 | 요구사항 해석이 2개 이상 가능하고 자체 판단 근거가 없는 경우 |
| 범위 확대 유혹 | 현재 PR의 허용 수정 범위를 넘는 변경이 필요해 보이는 경우 |
| 외부 의존성 장벽 | 외부 API, 서비스, 라이브러리의 제약으로 진행 불가한 경우 |
| 리뷰 미통과 | 리뷰 게이트에서 합의된 조건을 충족하지 못한 경우 |
| blocker 발견 | 해결할 수 없는 기술적 장벽을 발견한 경우 |

### 중단 시 행동

1. 현재 상태를 기록한다 (무엇을 하고 있었고, 어디서 막혔는지).
2. 인간에게 보고한다 (중단 사유 + 제안 선택지).
3. 인간의 지시가 올 때까지 해당 작업을 진행하지 않는다.

## 7. 금지 조항

에이전트가 **절대 해서는 안 되는 것**.

| 금지 조항 | 위반 시 조치 |
|-----------|-------------|
| 단계 게이트 우회 | 인간 승인 없이 다음 단계로 자동 진입하면 즉시 중단하고 직전 단계로 롤백 |
| Source of Truth 밖 추측 진행 | 공식 문서에 없는 문서·구두 합의를 근거로 진행하면 중단 |
| 리뷰 미통과 PR을 전제로 사용 | 리뷰가 끝나지 않은 PR을 다음 작업의 입력으로 쓰면 중단 |
| blocker 은폐 | 발견한 blocker를 보고하지 않고 우회하면 해당 작업 전체 재검토 |
| 인간 소유 항목 임의 확정 | 목표·비목표·성공 기준 같은 인간 소유 항목을 에이전트가 단독으로 결정하면 반려 |

## 8. 리뷰 게이트 골격

[4] PARALLEL REVIEW에서 사용하는 다관점 리뷰 골격이다. **자동 점수 차단 게이트가 아니라 인간 판정의 참고값**으로 운용한다.

### 리뷰 역할

| 역할 | 관점 | 검토 항목 |
|------|------|----------|
| **CEO** | 스코프·premise·10-star | 무엇을 만들고 무엇을 자를지, 더 큰 제품 가능성 |
| **Engineering** | 아키·데이터·엣지 | 구조 정합성, 위험, 테스트 커버리지, 확장성 |
| **Design / UX** | 사용자·일관성 | UX 흐름, 일관성, hierarchy, slop 패턴 |

코드 PR 단위 리뷰가 필요한 [프로젝트명 기입]은 추가로 다음 역할을 정의해도 좋다.

| 역할 | 관점 |
|------|------|
| critic | 잘못된 가정과 실패 모드 반박 |
| verifier | 테스트 완전성과 evidence 감사 |

### 점수 운용

각 역할은 0~10 척도로 정성 평가를 남긴다. 점수는 **인간 판정의 참고값**일 뿐, 자동 차단 임계값이 아니다.

| 점수 | 의미 |
|------|------|
| 9-10 | 완전함. 엣지 케이스 커버, evidence 충분 |
| 7-8 | 핵심 커버, 일부 누락이나 운영에 지장 없음 |
| 5-6 | 중요한 누락 존재, 수정 검토 |
| 1-4 | 형식적이거나 근본적 결함, 재작성 검토 |

세 리뷰가 상반되면 자동 우선순위는 없다. 인간이 결정자다.

### 결과 분기

| 결과 | 다음 행동 |
|------|----------|
| OK | [5] DECOMPOSITION 진입 |
| 수정 요청 | 가장 약한 영역 피드백 반영 후 PRD 갱신 → 재리뷰 |
| 중단 | 사유를 산출물 파일에 메모로 남김 |

### 재리뷰 에스컬레이션

- 수정 후 재리뷰 시 **모든 역할**이 다시 채점한다.
- 재작업 합산 **3회 연속**으로 수정 요청이 나오면 즉시 중단하고 인간에게 에스컬레이션한다.

### PR 단위 적용 (코드 PR)

[6] EXECUTION의 PR 단위 ralph loop 안에서도 위 리뷰 역할 일부를 적용한다. 핫픽스는 critic만 사전 적용 후 사후 48시간 내 전체 리뷰 보완을 권장한다.

## 9. PR 분해 기준

### PR 크기 기준

| 기준 | 권장 | 상한 | 초과 시 |
|------|------|------|---------|
| 변경 파일 수 | 1-5개 | 10개 | 분할 필수 |
| 변경 라인 수 (추가+삭제) | 50-200줄 | 400줄 | 분할 필수 |
| 변경 서비스 수 | 1개 | 1개 | cross-cutting 명시 시에만 예외 |
| 문서 전용 PR 라인 수 | — | 600줄 | 논리적 단위 분리 원칙 유지 |

자동 생성 파일(`package-lock.json`, `Package.resolved` 등)은 라인 수 계산에서 제외한다.

### 분할 불가 예외 사유 (PR 본문에 명시 필수)

| 사유 | 예시 |
|------|------|
| 원자적 변경 | 타입 변경이 10개 이상 파일에 연쇄 영향 |
| 마이그레이션 | 스키마 변경 + 코드 적응이 분리 불가 |
| 초기 스캐폴딩 | 새 서비스의 최초 디렉토리/설정 구조 생성 |

### Sprint 기준

| 기준 | 권장 | 상한 |
|------|------|------|
| sprint당 PR 수 | 3-5개 | 8개 |
| sprint 기간 (wall-clock, 리뷰 대기 제외) | 1-3일 | 5일 |

### 하나의 PR에 묶지 말아야 하는 것

| 금지 조합 | 사유 |
|-----------|------|
| 리팩토링 + 기능 추가 | 롤백 시 분리 불가 |
| 여러 서비스 동시 변경 | 배포 영향 범위 확대 |
| 버그 수정 + 기능 추가 | 롤백 시 버그 수정까지 되돌려짐 |
| 의존성 업그레이드 + 코드 변경 | 문제 원인 추적 어려움 |

## 10. gstack 스킬 라우팅

[프로젝트명 기입]은 단계마다 gstack 스킬을 호출해 작업한다. 아래는 표준 매핑이다. 프로젝트별로 추가/대체 스킬이 있으면 `CHARTER.md`에서 확장한다.

| 단계 | 기본 스킬 | 비고 |
|------|----------|------|
| [1] INTAKE | (없음 — 메타 룰) | 텍스트 길이·키워드 신호로 자동 분기 |
| [2] DISCOVERY | `/office-hours` | 실수요·문제 정의, 1:1 |
| [3] PRD DRAFT | `/brainstorming` | 한 번에 한 질문, 1:1 |
| [4] PARALLEL REVIEW | `/plan-ceo-review`, `/plan-eng-review`, `/plan-design-review` | 단일 메시지에 동시 dispatch |
| [5] DECOMPOSITION | `/writing-plans` | 단일 에이전트가 일관된 분해 |
| [6] EXECUTION | `ralph-loop`, `/ship`, `/qa`, `/review` | PR 단위 자율 루프 + N PR 병렬 |

### 보조 스킬

| 용도 | 스킬 |
|------|------|
| 시각 QA | `/design-review`, `/qa` |
| 출시 후 모니터링 | `/canary` |
| 회고 | `/retro`, `/document-release` |
| 디버깅 | `/investigate` |

## 11. 용어 사전

| 용어 | 정의 |
|------|------|
| PRD | Product Requirements Document. 인간이 작성하고 [3]에서 13필드 (핵심 11 + 시각화 2)로 정리되는 문서 |
| brief | DISCOVERY 단계에서 만들어지는 1쪽 요약 (Goal / Non-Goals / Success Criteria 후보) |
| 승인 게이트 | 단계 사이에서 인간이 OK / 수정 / 중단을 결정하는 지점 |
| ralph loop | 한 PR 안에서 구현→검증→수정을 자율 반복하는 미니 루프 패턴 |
| evidence trail | 검증 수행 결과를 구조화하여 기록한 증거 |
| stop rule | 에이전트가 즉시 중단해야 하는 조건 |
| harness | 에이전트의 실행을 안전하게 제약하는 시스템·스캐폴딩·규칙의 총체 |
| CEO 리뷰 | 스코프·premise·10-star 관점 리뷰 |
| Eng 리뷰 | 아키·데이터·엣지·테스트 관점 리뷰 |
| Design 리뷰 | UX·일관성·hierarchy 관점 리뷰 |
| critic | 잘못된 가정과 실패 모드를 찾는 반박자 (PR 단위 리뷰에서 활용) |
| verifier | 테스트 완전성과 evidence를 감사하는 검증자 (PR 단위 리뷰에서 활용) |

## 12. 프로젝트 적용 시 체크리스트

새 프로젝트가 이 템플릿을 가져갈 때 채워야 하는 항목.

### 필수 정의 항목

```
[ ] 프로젝트명 / 저장소 URL
[ ] 서비스 목록 및 각 서비스의 target path
[ ] 서비스별 검증 명령어 (예: cd service-a && npm run verify)
[ ] 산출물 저장 경로 (brief / PRD / 리뷰 / 분해 plan / evidence)
[ ] Source of Truth 문서 목록 (CHARTER, AGENTS, ARCHITECTURE 등)
[ ] 브랜치 전략 (base branch명, 브랜치 네이밍 규칙)
[ ] 배포 환경 및 명령어 (staging, production)
[ ] 이슈 트래커 종류 및 라우팅 규칙 (라벨 → target path)
```

### 권장 정의 항목

```
[ ] 스택별 추가 금지 조항 (예: 특정 라이브러리 사용 금지)
[ ] 프로젝트별 Stop Rule 추가 항목
[ ] 핫픽스 승인권자
[ ] gstack 스킬 외 보조 도구 (CodeRabbit 등) 연동 정책
```

### 이 문서에서 그대로 사용하는 것 (수정 불필요)

- 1~4장: 핵심 원칙, 소유권 모델, 멀티 에이전트 원칙, 사전 리뷰 조건 수집
- 5장: 6단계 실행 경로 (단계 이름과 게이트 패턴)
- 6~7장: Stop Rules, 금지 조항
- 8장: 리뷰 게이트 골격 (점수는 참고값으로 사용)
- 9장: PR 분해 수치 기준
- 11장: 용어 사전
