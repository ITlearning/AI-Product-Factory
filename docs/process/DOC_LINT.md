# Doc Lint

이 문서는 문서 전용 변경에 대한 자동 검증 규칙을 정의한다.
코드 변경 없이 문서만 수정한 PR도 검증 게이트를 통과해야 병합 승인을 요청할 수 있다.

---

## Purpose

문서는 하네스의 핵심 아티팩트다. 코드와 동일한 수준의 검증을 받아야 한다.

- 코드 PR은 `npm run verify`로 검증한다. 문서 PR은 이 문서의 규칙으로 검증한다.
- 검증 없이 문서 PR을 병합하지 않는다 (RELIABILITY.md 핵심 원칙).
- 자동화 가능한 검증은 자동화하고, 자동화 불가능한 검증은 수동 체크리스트로 대체한다.

---

## Execution Path 상의 위치

```
에이전트가 문서 PR 작성 → [doc-lint 검증] → 멀티 에이전트 리뷰 → 인간 병합 승인
```

doc-lint 검증은 멀티 에이전트 리뷰 이전에 실행한다. 검증 실패 시 리뷰를 시작하지 않는다.

---

## 검증 항목

### 1. 마크다운 구문 검증

| 규칙 | 설명 |
|------|------|
| 제목 계층 | `#`부터 순차적으로 사용한다. `##` 없이 `###`를 사용하지 않는다 |
| 테이블 구문 | 모든 테이블은 헤더 행 + 구분 행 + 데이터 행 구조를 충족한다 |
| 코드 블록 | 열린 코드 펜스(```)는 반드시 닫는다 |
| 빈 링크 | `[text]()` 또는 `[](url)` 형태의 빈 링크가 없어야 한다 |
| 후행 공백 | 줄 끝 불필요한 공백을 허용하지 않는다 (의도적 줄바꿈 `  ` 제외) |

### 2. 링크 유효성 검증

| 규칙 | 설명 |
|------|------|
| 내부 링크 존재 | 마크다운 링크 `[text](path)`의 대상 파일이 실제로 존재해야 한다 |
| 상대 경로 정확성 | 링크의 상대 경로가 해당 파일의 위치에서 올바르게 해석되어야 한다 |
| 앵커 유효성 | `#section` 앵커가 포함된 링크는 대상 문서에 해당 헤딩이 존재해야 한다 |
| 외부 링크 | 외부 URL은 검증 대상에서 제외한다 (네트워크 의존성 배제) |

### 3. 교차 참조 정합성 검증

| 규칙 | 설명 |
|------|------|
| Cross-References 테이블 | 모든 process 문서(`docs/process/*.md`)와 공통 가이드(`docs/*.md`)는 `## Cross-References` 또는 `## 관련 문서` 섹션을 가져야 한다 (한국어 문서는 후자를 사용해도 동일하게 인정) |
| 양방향 참조 | 문서 A가 문서 B를 Cross-References에 포함하면, 문서 B도 문서 A를 포함해야 한다 |
| Source of Truth Map 동기화 | `docs/process/CHARTER.md`의 Source of Truth Map에 등록된 모든 문서가 실제로 존재해야 한다 |
| AGENTS.md 동기화 | `AGENTS.md`의 Source of Truth 테이블에 등록된 모든 문서가 실제로 존재해야 한다 |
| 링크 형식 통일 | Cross-References 테이블의 문서 참조는 마크다운 링크 형식을 사용한다. plain text 경로를 허용하지 않는다 |

### 4. 필수 섹션 검증

process 문서(`docs/process/*.md`)와 공통 가이드(`docs/*.md`)는 다음 섹션을 포함해야 한다.

| 필수 섹션 | 설명 |
|-----------|------|
| `## Purpose` | 문서의 목적 |
| `## Cross-References` (또는 `## 관련 문서`) | 관련 문서 참조 테이블 |
| `## Change Log` | 변경 이력 |
| `## Fail-Closed 기본값` | fail-closed 규칙 (해당하는 경우) |

예외:
- `AGENTS.md`, `ARCHITECTURE.md`는 루트 문서이므로 이 규칙을 적용하지 않는다.
<!-- WORKFLOW.md 예외 행 제거: M-3에서 WORKFLOW.md 자체가 삭제됨 -->
- `docs/plans/*.md`는 계획 문서이므로 이 규칙을 적용하지 않는다.

### 5. Glossary 용어 일관성

| 규칙 | 설명 |
|------|------|
| 공식 용어 사용 | `docs/process/CHARTER.md`의 Glossary에 정의된 용어는 모든 문서에서 동일한 표기로 사용한다 |
| 용어 변형 금지 | Glossary 정의와 다른 표기를 사용하지 않는다 (예: "INTAKE" vs "intake", "PRD DRAFT" vs "PRD draft") |
| 신규 용어 | Glossary에 없는 전문 용어를 도입하면 Glossary 추가를 제안한다 |

### 6. Change Log 정합성

| 규칙 | 설명 |
|------|------|
| 변경 기록 | 문서를 수정한 PR은 해당 문서의 Change Log에 항목을 추가해야 한다 |
| 날짜 형식 | `YYYY-MM-DD` 형식을 사용한다 |
| PR 번호 | 해당 변경이 포함된 PR 번호를 기록한다 |

## 검증 방법

### 자동 검증 (향후 구현)

현재 Sprint 0에서는 자동화된 문서 린트 스크립트가 존재하지 않는다. 향후 다음 도구를 도입할 수 있다:

| 도구 | 용도 |
|------|------|
| markdownlint | 마크다운 구문 검증 (항목 1) |
| 커스텀 스크립트 | 링크 유효성, 교차 참조 정합성, 필수 섹션 검증 (항목 2-4) |
| 커스텀 스크립트 (향후) | Glossary 용어 일관성 검증 (항목 5) |
| 커스텀 스크립트 (향후) | Change Log 정합성 검증 (항목 6) |

자동화 도구 도입 시점은 인간이 결정한다. 도입되면 검증 명령을 `AGENTS.md` Validation 테이블에 추가한다.

### 수동 검증 (현재)

자동화 도구가 없는 동안, 에이전트는 다음 체크리스트를 수동으로 수행한다.

#### 문서 PR 검증 체크리스트

- [ ] 마크다운 구문: 제목 계층 순차, 테이블 구조 정상, 코드 블록 닫힘
- [ ] 내부 링크: 모든 `[text](path)` 링크의 대상 파일 존재 확인
- [ ] 상대 경로: 파일 위치 기준 상대 경로 정확성 확인
- [ ] Cross-References: 변경된 문서의 참조 테이블이 최신 상태인지 확인
- [ ] 양방향 참조: 새로 추가한 참조가 대상 문서에도 반영되었는지 확인
- [ ] Source of Truth Map: 새 문서 추가 시 CHARTER.md와 AGENTS.md에 등록 확인
- [ ] 필수 섹션: Purpose, Cross-References, Change Log 존재 확인
- [ ] Glossary 용어: CHARTER.md Glossary 정의와 일치하는 표기 사용 확인
- [ ] Change Log: 변경 사항이 해당 문서의 Change Log에 기록되었는지 확인

에이전트는 이 체크리스트의 수행 결과를 evidence-trail 형식으로 기록한다.

---

## 검증 범위

| 규칙 | 설명 |
|------|------|
| 변경 범위 한정 | 변경된 문서와 그 문서가 참조하는 문서만 검증한다. 관련 없는 문서는 검증하지 않는다 |
| 신규 문서 | 새 문서 추가 시 해당 문서의 전체 항목을 검증한다 |
| 교차 참조 영향 | Cross-References나 Source of Truth Map을 변경한 경우, 영향받는 모든 문서의 참조 정합성을 검증한다 |
| 혼합 PR (코드 + 문서) | 코드와 문서를 동시에 변경하는 PR은 코드 검증(`npm run verify`)과 doc-lint 검증을 모두 수행한다. 문서 부분에 대해 이 문서의 모든 규칙이 동일하게 적용된다 |
| 루트 문서 | `README.md` 등 루트 레벨 문서는 필수 섹션 규칙(항목 4)의 적용 대상이 아니다. 링크 유효성(항목 2)은 적용한다 |

## 검증 실패 시 행동

| 상황 | 행동 |
|------|------|
| 자동 검증 실패 | 실패 항목을 수정한다. 수정 불가 시 인간에게 보고한다 |
| 수동 검증 실패 | 실패 항목을 수정한다. 수정 불가 시 인간에게 보고한다 |
| 검증 실행 불가 | PR을 커밋하지 않는다 (RELIABILITY.md 기준) |
| 부분 검증 | 일부 항목만 검증 가능한 경우, 검증한 항목과 검증하지 못한 항목을 명시한다 |

## Fail-Closed 기본값

- 검증 미수행 문서 PR은 리뷰를 시작하지 않는다.
- 링크 유효성이 불확실하면 깨진 링크로 취급한다.
- 교차 참조 정합성이 불확실하면 불일치로 취급한다.
- 필수 섹션 존재가 불확실하면 누락으로 취급한다.
- 용어 일관성이 불확실하면 불일치로 취급한다.
- Change Log 업데이트가 불확실하면 미기록으로 취급한다.
- 자동 검증 도구와 수동 체크리스트의 결과가 충돌하면 더 엄격한 결과를 따른다.
